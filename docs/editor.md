# Editor & UI Reference

Implementation details for the code editor, layout, preferences, and UI conventions. See `CLAUDE.md` for core architecture.

## Error Display

Parse errors get a squiggly underline at the error location via `QTextCharFormat` with `SpellCheckUnderline` style, applied as an extra selection on the `QPlainTextEdit`. Also reported in the console.

## Find / Replace

`CodeEditor.show_find(replace=False)` opens a `FindBar` overlay parented to the editor, top-right corner. `show_find(replace=True)` also shows the replace row. Triggered by Cmd+F / Cmd+H.

`FindBar` features:
- Plain-text and regex search (`.*` toggle), case-sensitive toggle (`Aa`)
- All matches highlighted pale yellow; current match orange with white text
- Match count label ("N of M"); prev/next navigation (◀ ▶ or Shift+Enter / Enter)
- Replace one (current match) and Replace All (works backwards through matches to preserve positions, wrapped in one `beginEditBlock`/`endEditBlock` for one undo step)
- A single-word selection present when Find opens pre-populates the search field
- Escape closes the bar and returns focus to the editor
- `_find_selections` is a separate extra-selection list on `CodeEditor`, inserted between `_selection_extra` and `_exec_selection` in `_refresh_extra_selections`
- Document changes while open auto-rerun the search via `document().contentsChanged`

## Indent Guides

Faint vertical lines drawn inside each indented line's leading whitespace, every `_indent_size` columns, except at the column of the first non-whitespace character. Implemented as `_IndentGuides(QWidget)`, a transparent overlay on `CodeEditor.viewport()`, created before `_ColumnGuide` so the column guide renders on top. Repainted on `document().contentsChanged` and `set_indent_size()`.

Both this and the Column Guide overlay are viewport-child widgets whose own geometry has to stay pinned to `(0, 0, viewport.width(), viewport.height())` — `CodeEditor._reposition_scroll_overlays()`, connected to both `horizontalScrollBar().valueChanged` and `verticalScrollBar().valueChanged` (not just `resizeEvent`, which was the only place `update_geometry()` was called before this was added), re-asserts that on every scroll. Without it, `QPlainTextEdit`'s internal scroll optimization moves the viewport's own child widgets by the scroll delta as a side effect (confirmed empirically: scrolling to hbar value 100 left `_column_guide.geometry()` at `x=-100`) — since each overlay's `paintEvent` already recomputes its line's position fresh from `cursorRect()` (itself already scroll-adjusted), a widget origin that *also* drifted by the same delta doubles the effective on-screen movement per scroll step, so the guide moved twice as fast as the actual text and never stayed aligned with its column.

Paint logic: for each visible block, count leading spaces `n`; draw guides at `indent_size, 2*indent_size, …` while `col < n` (strictly less, so the column at `n` is never drawn). Empty/unindented lines skipped. Uses `QFontMetricsF` for sub-pixel accuracy.

Both this and the Column Guide draw their line(s) via the module-level `_draw_vline_avoiding_cursor(painter, x, y_top, y_bottom, cursor_rect)` rather than a plain `painter.drawLine()`: since both overlays are raised viewport-child widgets, their solid guide pixels paint *over* the text edit's own cursor (a child can't be told to render behind its own parent's content, only reordered among siblings) — a guide line landing on the caret's column would otherwise hide it completely. The helper splits the line into up to two segments stopping exactly at `editor.cursorRect()`'s top/bottom edges when `x` falls within the cursor rect's horizontal span, leaving its full interior undrawn; when `x` doesn't overlap the cursor at all, it's a single unmodified `drawLine()`.

## Column Guide

A faint vertical line at column 80, implemented as `_ColumnGuide(QWidget)`, a transparent overlay on `CodeEditor.viewport()`:
- `WA_TransparentForMouseEvents` + `WA_TranslucentBackground` so only the line pixel shows and mouse events pass through
- `update_geometry()` keeps the overlay sized to the full viewport rect; called from `CodeEditor.resizeEvent()` and `_reposition_scroll_overlays()` (see Indent Guides above for why the latter is needed)
- x position = `cursorRect(cursor_at_pos_0).x() + QFontMetricsF(font).horizontalAdvance('0' * 80)`. `QFontMetricsF` (not `QFontMetrics`) is required — the integer version rounds character width up by ~0.2px, accumulating to ~2 columns of error over 80 characters.

## Code Folding

Fold markers (▼ unfolded, ▶ folded) appear in the right section of the line-number gutter; clicking calls `toggle_fold(block_number)`.

`_compute_fold_regions(doc)` returns `{open_block: close_block}` via two passes:
1. **Delimiter matching** — `{…}`, `(…)`, `[…]` pairs; a region only forms when opener and closer are on different lines
2. **Indentation continuation** — any non-empty line followed by at least one more-indented non-empty line; covers function bodies, ternary chains, nested list comprehensions, etc.; `setdefault` lets pass-1 delimiter regions take precedence

`_fold_regions` recomputes lazily on first paint after `_fold_dirty` is set by `_on_doc_changed`. `_fold_busy` guards against re-entrant recomputation and against `_on_doc_changed` resetting `_fold_dirty` mid-toggle.

`_set_range_visible(start_bn, end_bn, visible)` sets `QTextBlock.setVisible()` on each hidden block, then a no-op `cursor.beginEditBlock(); cursor.endEditBlock()` forces `QPlainTextDocumentLayout` to recalc block heights (required for visibility changes to take effect).

Fold indicators are drawn with `painter.drawPolygon(QPoint[])` — `QPainterPath.drawPath` was invisible at small sizes on macOS; `drawPolygon` is reliable.

## Editor Context Menu

Right-clicking in the editor builds a standard Qt context menu, then appends identifier-aware and debug-aware actions.

**Debug variable inspection** (when debugger is paused and the word under the cursor is a known variable — locals, globals, constants, or `$`-specials):
- **`Variable: x`** / **`Value: <value>`** — two disabled (grayed-out) header items: the variable name and its value formatted by `_fmt()` and truncated to 30 characters with `…` if longer. Appear before the standard cut/copy/paste items, followed by a separator.
- **Print 'x' to Console** — emits `CodeEditor.print_value_to_console(name, value)`, connected to `MainWindow._on_debug_print_value`, which calls `self._console.append_value(name, value, _pretty_assignment(name, value))`. The original Python value is stored for the console right-click viewer menu.
- **View 'x'…** submenu — populated by `build_viewer_menu()` from `data_viewers.py`; only appears when the value type supports a viewer (list, VNF, path, grid, matrix, affine transform).

The available variables come from the innermost debug frame: `{**outer_scope, **local_scope}` (local overrides outer on collision), which covers locals, globals, constants, and `$`-specials. `Qt.WordUnderCursor` excludes `$`, so `contextMenuEvent` manually checks whether the character immediately before the selection is `$` and prepends it — allowing `$fn`, `$t`, etc. to match.

`MainWindow._on_debug_paused` and `_on_debug_error_break` call `_set_debug_locals_on_visible(locals_dict)`, which first clears locals from all editors via `_clear_all_debug_locals()`, then sets them on `_current_tab().editor` — the editor that is actually visible after `_show_debug_line` may have switched to an included file's tab. All resume/step/stop/finish/restart handlers call `_clear_all_debug_locals()` to sweep every open editor clean.

**Go to Definition** (for any identifier, always shown):

Right-click an identifier shows "Go to Definition of 'name'", only for words matching `\$?[A-Za-z_][A-Za-z0-9_]*` (plain identifiers and `$`-prefixed specials).

`CodeEditor.go_to_definition_requested` (emits the word) connects to `MainWindow._go_to_definition(tab, word)` per tab.

`_go_to_definition` requires a cached `root_scope` on the tab (set after every successful `build_scopes()`, both in the render worker and `_start_debug()`); if absent, logs a message asking the user to render first.

Lookup order: `scope.lookup_variable(word)` → `scope.lookup_function(word)` → `scope.lookup_module(word)`, first non-None wins. Built-in modules return `None` from `lookup_module` and are skipped.

The definition node's `.position.origin` gives the source file path, `.position.line` the 1-indexed line. Navigation uses `scroll_to_line(line)` which ensures at least 5 lines of context above and below:
- Same file (or origin `None` / untitled tab): scroll current editor to the line
- Different file: switch to a matching open tab by `file_path`, or open via `_create_and_add_tab()` followed by `_render(tab)`

`_create_and_add_tab(path, text) -> FileTab` creates a `FileTab` (editor + file metadata) and adds it to `_tabs` in the editor dock. Also registers the editor with `DocumentManager` for cross-window sync. If the only existing tab is an empty, unmodified Untitled tab, it is replaced rather than kept alongside. Used by `_open_file`, `_open_recent`, `_go_to_definition`, `_find_or_open_tab`; not by `_new_document` (different setup path for blank tabs).

**Read-only library files**: `_create_and_add_tab` also sets `tab.editor.setReadOnly(True)` when the opened path resolves under the OpenSCAD library directory (`library_manager._library_dir()` — e.g. installed BOSL2 files), so files opened via "Use Library" or "Go to Definition" into a library aren't edited by accident. `FileTab.display_name()` appends `" (ro)"` to the tab label whenever `editor.isReadOnly()` is true (alongside the `"*"` modified marker). The Edit ▸ Read Only checkable action reflects and toggles the current tab's state (synced in `_tab_changed`), letting library developers make any tab editable again.

**Lexical "View as..."/"Edit as..."** (for a plain numeric literal under the cursor, independent of debug session state — unlike the debug-gated "View 'x'…" submenu above, which is keyed by identifier name and requires a paused debug session):

`contextMenuEvent` locates the enclosing `[...]` literal(s) at the click offset via `data_viewers.find_viewable_literals`/`find_editable_literals` (both plural), each backed by the shared `_iter_enclosing_literals` generator: it bracket-matches outward from the cursor, `ast.literal_eval`s each candidate span, and skips (but keeps walking past) any level that fails to parse — i.e. contains identifiers, calls, or OpenSCAD-only syntax like ranges or `true`/`false`/`undef`. This also means "numeric-only, no expressions" falls out of `ast.literal_eval` for free, with no extra grammar checking needed.

- **View as...** — `find_viewable_literals` (plural) matches each of List/VNF/Grid/Path/Region *independently* — a separate walk-outward search per shape, not one shared "first match wins" walk — returning `{shape: (start, end, value)}` for whichever match. This independence matters because `_is_list` is trivially true for any list: a shared walk would almost always resolve at the innermost bracket (e.g. clicking anywhere inside an outer path `[[0,0],[1,0]]` usually lands inside one inner point's own brackets), permanently starving "View as Path..." of a reachable click position — a real bug this shape caused, not just a cosmetic quirk. Each shape now finds its own appropriate level: "list" still resolves to the inner point, while "path" keeps walking out to the enclosing path, so both options show together. `build_lexical_view_menu` builds one action per matched shape (each from its own span/value), reusing the existing `_open_list_viewer`/`_open_vnf_viewer`/`_open_grid_viewer`/`_open_path_viewer` unchanged, non-modal. Works on read-only tabs too, since viewing doesn't mutate anything.
- **Edit as...** — `find_editable_literals` (plural, same independent-per-shape search as `find_viewable_literals`) matches each of Path/Grid/Matrix/Affine/VNF/Region independently, as `{shape: (start, end, value)}`. This independence matters here too: a grid's own row is itself a valid Path (a list of numeric points), so a single shared "first match wins" walk would resolve "path" (the row) before ever reaching "grid" (the whole structure) for *any* click inside a row — a real bug caught while writing a grid test file, not just a theoretical edge case. `build_editor_menu` builds one action per matched shape (each from its own span/value — its `on_commit(new_text, start, end)` closure is bound per-shape so Save writes back to the *right* span), opening the same viewer classes in a new **editable** mode (`editable=True`): cell edits (`itemChanged`) only update the dialog's own in-memory value and live preview (via the existing `load_path`/`load_grid`/`load_matrix` refresh hooks) — no source writeback, no render. An explicit **Save** button serializes the whole edited value via `data_viewers._format_value` — which builds a synthetic `ListComprehension`/`NumberLiteral` AST (with a placeholder `Position`), wraps it in a throwaway `Assignment` (`_v = <value>;`), and renders that through `openscad_lalr_parser.to_openscad`, then strips the synthetic `"_v = "`/`";"` back off — the same "build synthetic AST, pretty-print it" pattern `debugger._pretty_assignment` already uses for formatting debug values, rather than hand-rolled string joining. The `Assignment` wrapper is required, not cosmetic: `to_openscad` only routes a node through the real width-aware formatter (which decides inline vs. multi-line list layout, wrapping past 80 chars) for specific statement types — a bare expression node falls through to a naive `str(node)` with no line-wrapping at all. `_format_value` emits the result through the dialog's `committed` signal exactly once; **Cancel** discards everything. The dialog is **non-modal** (`.show()`, same as the read-only viewers) — not modal at all. Modal was tried first (both `setModal(True)`/`ApplicationModal` and `setWindowModality(Qt.WindowModality.WindowModal)`) to block editing the parent editor's source while the dialog is open, but *any* Qt-modal dialog turned out to suppress the main window's View-menu shortcuts for the dialog's own embedded viewport, regardless of modality flavor (Qt suppresses non-active-window `WindowShortcut`/`ApplicationShortcut`-context actions whenever any modal widget is active — see `docs/editor.md`'s "Keyboard shortcuts" note below). So instead, `data_viewers._lock_parent_editor_while_open(dlg, parent)` sets `parent.setReadOnly(True)` when the dialog opens and restores it (`False`) on the dialog's `finished` signal — achieving the same "don't let the source change out from under the tracked span" goal without touching Qt modality at all. Skipped entirely on read-only tabs.

`CodeEditor.replace_span(start, end, new_text)` performs the actual source-text write-back when Save is clicked, then emits `source_edited_externally`, connected per-tab (in both `_create_and_add_tab` and `_new_document`) to `self._render(tab)` — the same "commit → render" pattern as a gizmo drag's mouse-up commit. `replace_span` doesn't need any custom undo handling: native Qt undo is disabled on `CodeEditor` (`setUndoRedoEnabled(False)`), and any document change already flows through `MainWindow._on_editor_changed` (connected to `document().contentsChanged`), which auto-captures a `_TextEditCmd` onto the app's own `QUndoStack` — exactly as it does for ordinary typing.

**Dialog title for lexically-launched viewers/editors**: the `title` string these dialogs are constructed with used to be a 30-char raw-text preview of the literal's own source (e.g. `text[start:end][:30]`). For a multi-line literal this was nearly useless — a window/menu title can't render an embedded newline, so a title like `"[\n    [0, 0], ..."` visually truncates to just `"["` (the reported symptom: a dialog titled `Grid Editor: [`). Instead, `_literal_display_name(text, start)` looks backward (up to 200 chars) via `_ASSIGNMENT_NAME_RE` for a simple `name = ` immediately preceding the literal's start, and returns `name` if found. `build_lexical_view_menu`/`build_editor_menu`'s `_preview` helpers call this instead of slicing raw source text. Each viewer's `setWindowTitle` then omits the `": {title}"` suffix entirely when the name lookup comes back empty (e.g. an inline literal like `translate([10,20,30])`'s argument, which isn't a `name = ` assignment) — the title is just `"Grid Editor"` rather than a colon dangling with nothing useful after it.

**Closure bug caught by manual testing on `test26.scad`**: `build_lexical_view_menu`/`build_editor_menu` each build several `QAction`s in a row from the same local `start`/`end`/`value` variables (reassigned once per matched shape, e.g. once for `"path"`, once for `"grid"`). The outer per-action lambdas originally closed over those variable *names* rather than their values at the time each action was built (Python closures capture by reference, not by value) — since all the `if` blocks run immediately while the menu is being constructed, every action ended up reading whichever shape's `start`/`end`/`value` was assigned *last*, regardless of which menu item was actually clicked. Concretely: right-clicking inside a grid row in `test26.scad` (where both "Edit as Path..." and "Edit as Grid..." match, from independent spans) and choosing "Edit as Path..." opened `PathViewer` with the *grid's* nested-list value instead of the row's flat point list, crashing in `_make_vert_table` (`f"{p[j]:g}"` — `p[j]` was itself a point-list, not a number). Fixed by binding `start=start, end=end, value=value` as default arguments on each outer lambda, forcing early (per-action) capture — the same fix already applied to the inner `on_commit` lambdas' `s=start, e=end` binding, just missing from the outer ones.

## Code Completion

`QCompleter` with a `QStringListModel` provides prefix-based autocomplete. The popup appears after 2+ identifier characters are typed and hides when there are no matches or an exact match.

**Word list**: 96 built-in names (keywords, modules, functions, constants, and `$`-variables like `$fn`/`$fa`/`$fs`/`$t`/`$children`/`$vpt`/`$vpr`/`$vpd` from `_BUILTIN_WORDS`) plus user-defined names (including user `$`-variables) from the last successful `build_scopes()`. `update_user_names(scope)` extracts names from `scope.variables`, `scope.functions`, and `scope.modules` dicts; called from `_RenderCallback.on_ast_ready` after each successful parse.

**Key handling in `keyPressEvent`**: when the popup is visible, Enter/Tab accept the selected completion and Escape dismisses. Tab for indent only fires when the popup is not visible. Other keys pass through to the editor and then `_update_completer_popup()` refreshes the prefix/popup state.

`_text_under_cursor()` walks backward from the cursor to find the current identifier prefix (alphanumeric + underscore + leading `$`). `_insert_completion()` replaces the prefix with the selected completion.

## Undo/Redo

Code edits and gizmo drags are undo/redo-able via Qt's `QUndoStack`. Each operation is a `QUndoCommand` subclass:

- **Code edits**: `TextEditCommand` stores before/after document state and calls `QPlainTextEdit.setPlainText()` on undo/redo
- **Gizmo ops**: `GizmoCommand` stores before/after source text and re-triggers a render on redo

All Cmd+Z / Cmd+Shift+Z route through `QUndoStack`, which disables `QPlainTextEdit`'s built-in undo (`setUndoRedoEnabled(False)`).

## Console Output

The console is `ConsoleWidget` (`window/console.py`), a `QTextBrowser` subclass. One instance exists per `MainWindow` (window-level singleton `self._console`) — there is no per-tab console. Plain log output goes through `MainWindow.log(text)` or `MainWindow.log_to_tab(tab, text)` (tab argument is ignored) → `self._console.append_output(text)`. Value-bearing output (from "Print to Console" or debugger return values) goes through `MainWindow.log_value_to_tab(tab, name, value)` or directly in `_on_debug_print_value` → `self._console.append_value(name, value, _pretty_assignment(name, value))`.

`QTextBrowser` is used (not `QPlainTextEdit`) because it handles cursor shapes natively: `PointingHandCursor` over anchor links, `IBeamCursor` over selectable text. `setOpenLinks(False)` prevents navigation; `anchorClicked` handles fold toggles.

`append_output(text)` and `append_value(name, value, text)` both route:
- **Single-line text** → `_append_plain(text)` — inserts a plain text paragraph via `QTextCursor`.
- **Multi-line text** → `_append_foldable(lines[0], '\n'.join(lines[1:]))` — inserts the first line as an HTML anchor `<a href="fold:N">▼ summary</a>` and the remaining lines as plain paragraphs. Clicking the anchor collapses or expands the block; `QTextBrowser` shows a hand cursor over it automatically.

`append_value` additionally stores `(name, value)` in `_fold_values: dict[int, tuple[str, object]]` keyed by fold_id, so right-clicking on any line of that block can launch a viewer.

Fold state is tracked in `_fold_headers: dict[int, tuple[int, int, int]]` (fold_id → (header_bn, first_body_bn, last_body_bn)) and `_folded: set[int]` (fold_ids currently collapsed). `clear()` resets all three dicts. Block visibility is toggled with `QTextBlock.setVisible()`; layout recalculation is triggered by updating the arrow character via `QTextCursor.insertText` (done after setting visibility so the `documentChanged` signal sees the correct state). The fold_id is a sequential integer assigned at append time and embedded in the anchor href.

**Console right-click context menu** (`MainWindow._console_context_menu`): `ConsoleWidget.value_at(pos)` checks whether the right-clicked position falls inside a fold header or body block that has a stored value (by scanning `_fold_headers` for block number, then looking up `_fold_values`). If found, a **View 'name'…** submenu is added via `build_viewer_menu()` before the **Clear Console** action.

The console displays:
- Parse errors (file/line/col from AST metadata)
- On each render: bounding box of the resulting mesh and current camera position

## Animation

The **Animate** dock (`AnimatePane` in `window/animate.py`, one per `MainWindow`) implements OpenSCAD's [`$t` animation](https://en.wikibooks.org/wiki/OpenSCAD_User_Manual/Animation):

- **Time / FPS / Steps** fields: Time shows the current `$t` (read-only display, but editable — typing a value jumps to the nearest step, clamped to `[0, 1 - 1/steps)`); FPS (1-1000) sets the playback rate; Steps (1-1,000,000) sets the number of frames in one cycle. `$t = step / steps` for `step` in `0..steps-1`. Tab/Shift+Tab move between these three fields, and Enter confirms an edit (`QLineEdit.editingFinished`, which fires on Return as well as focus-out). `AnimatePane` installs an event filter on each field to accept the `ShortcutOverride` event for Tab/Backtab — otherwise the main window's Indent/Undent actions (bound to Tab/Shift+Tab as window-wide shortcuts for the code editor) would consume the key before normal focus-navigation gets it.
- **Big play/pause button** and the **transport row** (First / Previous / Play / Pause / Next / Last) drive playback. Any non-playback transport action pauses playback first.
- **Dump Pictures** checkbox: when checked and Play is pressed, BelfrySCAD prompts (once per tab) for a destination folder via a folder picker, then saves each frame of one full animation cycle as `frameNNNN.png` (via `Viewport.grabFramebuffer()`), pausing automatically after frame `steps - 1` rather than looping.

Each frame change re-renders the current file with `$t` set accordingly — `MainWindow._viewport_params()` includes `"$t": self._animate_pane.current_t()`, merged into the evaluator's dynamic context alongside `$vpt`/`$vpr`/`$vpd` (see `docs/evaluator.md`). During playback the viewport camera is **not** auto-fit to the model's bounding box on each frame (unlike a normal Render), so the camera stays put across frames. Animation re-renders whichever file tab is currently visible in the editor.

## Customizer

`CustomizerPane` (`window/customizer.py`) mirrors OpenSCAD's Customizer: it scans the active source for top-level variable assignments with simple literal values and generates form widgets. Widget changes write back to the source immediately (no auto-render — user presses F6 to render).

**Parameter scanning** (`scan_parameters(source)`): reads lines top-to-bottom, tracking brace depth to skip nested scopes. A line comment (`// text`) immediately above an assignment becomes that parameter's description. Tab-group block comments (`/* [TabName] */`) group subsequent parameters; `/* [Hidden] */` suppresses all following parameters until the next group. Qualifying assignments have a name that is not a keyword or `$`-prefixed, and a value that parses as one of: `int`, `float`, `bool` (`true`/`false`), `"string"`, or a vector literal `[a, b, c, d]` of 1–4 numeric elements.

**Constraint syntax** (trailing `// comment` on the assignment line):

| Constraint | Widget | Notes |
|---|---|---|
| `[max]` | Slider | range 0…max |
| `[min:max]` | Slider | range min…max |
| `[min:step:max]` | Slider | explicit step |
| `[a, b, c]` | Dropdown | numeric or string values |
| `[a:Label, b:Label]` | Dropdown | values with display labels |
| `N` (integer) | Text field | max length N (strings only) |
| (none, bool) | Checkbox | |
| (none, number) | Spin box | step auto-derived from decimal places |
| (none, vector) | Row of spin boxes | one per element |

**Write-back** (`write_back_value(source, name, new_value)`): re-scans to find the parameter's line number, then does a regex substitution on that line to replace the value literal, preserving any trailing `// constraint` comment.

**Structural equality optimisation**: when the source changes but the set of parameters (names, tabs, descriptions, constraints) is unchanged, only widget values are updated — the full UI rebuild is skipped.

**Integration** (`main_window.py`): `_customizer_pane.set_source(text)` is called on tab switch and on every editor keystroke for the active tab. `source_changed` signal updates the editor text and sets the modified flag (preserving cursor position) without triggering a render.

## Application Preferences

Preferences live under the `editor/`/`viewport/` key groups in `QSettings("BelfrySCAD", "BelfrySCAD")`, accessed via `load_preference(key, type)` / `save_preferences(dict)` in `preferences.py`. `PreferencesDialog` is organized into **Editor** and **Viewport** tabs (`QTabWidget`) and has no OK/Cancel — only a **Close** button. Every control saves its value and calls the `on_change` callback (`MainWindow._apply_preferences`, passed in by `_open_preferences()`) immediately on change (`currentTextChanged`/`valueChanged`/`toggled` → `PreferencesDialog._emit(key, value)`), so edits are reflected live in every open tab/viewport rather than waiting for the dialog to close.

| Stereo depth scale | `viewport/stereoDepthScale` | `0.75` | Viewport |

`MainWindow._apply_preferences()` reads all settings and pushes them to every open tab via `_apply_preferences_to_tab(tab, font, indent, show_guide, guide_col)`, and to every open `Viewport` (color theme, IPD, screen distance, stereo depth scale). Called on startup (end of `_restore_settings()`), on every live preference change, and by `_new_document()`/`_create_and_add_tab()` so new tabs inherit current settings immediately.

**Color theme**: the Viewport section's "Color theme" dropdown (`preferences.py`) lists the named themes in `color_themes.COLOR_THEMES` — each a `{"background", "object", "axes"}` triple of RGBA tuples lifted from OpenSCAD's built-in render color schemes (`background`, `cgal-face-front`, and `axes-color`; "Cornfield" is a manual entry — OpenSCAD's classic default scheme — with `axes` hardcoded to black rather than taken from a JSON file). `_apply_preferences()` looks up the selected theme and sets `renderer.bg_color`, `renderer._default_color`, and `renderer.axes_color` on every open `Viewport`, including data-viewer dialogs, then calls `vp.update()` to repaint immediately. All three are resolved fresh every `paint()`/`_render_axes()`/`_get_label_texture()` call rather than baked into uploaded geometry: `MeshBuffer.color` stores the `ColoredBody`'s explicit `color()` override or `None`, and the draw loop substitutes `renderer._default_color` at draw time when it's `None`; `axes_color` only recolors the negative-axis lines/ticks and tick labels (the "gray" elements) — the positive X/Y/Z axis lines and ticks keep their fixed red/green/blue. Axis-label textures are cached by `(text, axes_color)` so a theme change invalidates stale-colored labels. All theme fields take effect on the next repaint without needing a full re-render.

`CodeEditor.set_indent_size(n)` stores `_indent_size` and updates `tabStopDistance`. Indent/unindent logic lives in `CodeEditor._indent_lines()`/`_unindent_lines()`, called both from `keyPressEvent` (Tab/Shift+Tab) and the Edit menu actions. Both handle single-line and multi-line selections.

**Auto-indent on Return**: the new line inherits the current line's indentation. An extra indent level is added when the current line ends with `{`, `[`, `(`, or starts with the keyword `function` or `module`. Typing `}`, `]`, `)` on a whitespace-only line removes one indent level.

**Down arrow on last line**: pressing Down on the last line appends an empty line and moves the cursor to it, so the user doesn't need to move to the end and press Return to extend the file.

The Preferences action uses `QAction.MenuRole.PreferencesRole` so Qt places it in the macOS application menu (Cmd+,).

**Word Wrap** is a checkable Edit menu item stored at the top level in `QSettings` under `"wordWrap"` (default `False`), not under `editor/`. Persisted/restored in `closeEvent`/`_restore_settings` using the same `blockSignals` pattern as `perspective`, and applied via `_apply_word_wrap_to_tab(tab)` (also called for new tabs in `_new_document` and `_create_and_add_tab`).

## Syntax Highlighting

`OpenSCADHighlighter(QSyntaxHighlighter)` applies keyword/type/number/string/comment formatting via regex rules processed per block in `highlightBlock()`.

Multi-line `/* ... */` block comments use `QSyntaxHighlighter` block state tracking (`previousBlockState`/`setCurrentBlockState`) so highlighting persists across lines. Both `//` line comments and `/* */` block comments render in the same green (`#6A9955`).

## Startup Behavior

Opens with a single blank untitled document. When opening a file while the only tab is an empty, unmodified "Untitled" tab, that tab is replaced by the new file tab (in `_create_and_add_tab`).

## File Opening

Files can be opened in several ways:

- **File > Open** — standard file dialog
- **File > Recent Files** — reopens previously opened files; stale entries are auto-pruned
- **Drag and drop** — `.scad` files dropped onto the window open as new tabs
- **macOS file association** — `.scad` files opened from Finder (double-click or Open With) send a `QFileOpenEvent` to `BelfrySCADApp`, which forwards to `MainWindow.open_file_by_path()`; the Info.plist declares `CFBundleDocumentTypes` for `.scad` via briefcase config in `pyproject.toml`
- **Command-line arguments** — `BelfrySCAD foo.scad bar.scad` opens each file at launch

All paths converge on `open_file_by_path(path)`, which resolves the path and checks for an already-open tab before creating a new one.

## Quit Behavior

When the user quits the app and there are modified editors open, a Save/Discard/Cancel dialog is shown for each unsaved tab. Cancel aborts the quit.

- **Console** (bottom dock): single running log per window; not per-tab
- **Status bar**: bottom strip; four camera labels displayed as `Viewport: translate=[…]  rotate=[…]  dist=…  FoV=…` (translate at 2 dp, others at 1 dp), each with a tooltip ("Viewport Translate ($vpt)", "Viewport Rotation ($vpr)", "Viewport Distance ($vpd)", "Viewport FOV ($vpf)") and a right-click context menu with **Copy $vp\*** (copies the single variable, e.g. `$vpt = [0.00, 0.00, 0.00]`) and **Copy all $vp\* values** (copies all four as semicolon-terminated OpenSCAD assignments, one per line); a viewport size label `(W × H)` in logical pixels (updated via `size_changed` signal from `resizeGL`); and an FPS counter. All four `$vp*` values are passed to the evaluator as OpenSCAD special variables via `_viewport_params()`. Default FOV is 22.5°. If the script assigns any `$vp*` variable, `_apply_vp_params()` updates the camera after render and skips the normal auto-fit; the same update fires on every debugger pause so the viewport tracks the script's camera state while stepping.

The editor, console, debugger, animate, and customizer panes are `QDockWidget` instances — dockable to any side or floatable, with position/visibility persisted via `QSettings("BelfrySCAD", "BelfrySCAD")` (`saveState()`/`restoreState()`). Object names: "EditorDock", "ConsoleDock", "DebuggerDock", "AnimateDock", "CustomizerDock". All four corners are owned by the side docks (left owns top-left/bottom-left, right owns top-right/bottom-right), so the bottom dock is constrained between them. `setDockNestingEnabled(True)` allows docks to be split within an area; `setAnimated(False)` works around a Qt crash in `QVariantAnimation` during drag-to-tab operations. On first launch (no saved `windowState` matching `_LAYOUT_VERSION`), `showEvent` fires a deferred `_set_default_layout` call: editor dock ≈ 30% width, right dock ≈ 25% width, bottom dock ≈ 17% height, right dock split 60/40 between debugger and customizer/animate. The Debugger pane is a single shared widget on `MainWindow` (not per-tab).

**Detached tab bar**: `_DetachedTabBar(QWidget)` (in `main_window.py`) is a `QTabWidget`-compatible facade splitting a plain `QTabWidget` into two independently-placed pieces: a `QTabBar` (`self.tab_bar`) and a `QStackedWidget` (`self._stack`, which *is* `_DetachedTabBar`'s own layout content). `MainWindow` puts `_stack` inside the Editor dock exactly where a real `QTabWidget` used to sit (`self._editor_dock.setWidget(self._tabs)` is unchanged), but places `tab_bar` in a separate, non-movable `QToolBar` ("TabBarToolBar") added via `addToolBarBreak()` right after the main toolbar — toolbar areas always span the full window width above the dock/central-widget region, unlike the Editor dock, which is only as wide as its own dock area. `_DetachedTabBar` implements only the `QTabWidget` methods `MainWindow` actually calls (`addTab`, `removeTab`, `widget`, `count`, `currentIndex`/`setCurrentIndex`, `currentWidget`/`setCurrentWidget`, `indexOf`, `setTabText`, `setMovable`, `setTabsClosable`, `setDocumentMode`, `tabBar`) plus `currentChanged`/`tabCloseRequested` signals forwarded from the internal `QTabBar` — so none of `MainWindow`'s ~60 other call sites needed to change. Dragging a tab to reorder it (`setMovable(True)`) fires `QTabBar.tabMoved`, which `_DetachedTabBar` mirrors into the stack via `removeWidget`/`insertWidget` so page order stays in sync with tab order. View ▸ Show Tab Bar toggles `_tab_bar_toolbar.setVisible` directly (not `tabBar().setVisible()`) so the whole strip collapses rather than leaving an empty toolbar row.

Scale markers are tick marks along the viewport axes showing distance units (Show Scale Markers), each labeled with its distance value. Labels are rendered in 3D as camera-facing textured billboards: each tick's number is rasterized to an RGBA texture (cached by string) and drawn on a small transparent quad positioned just past the tick, so labels respect depth (occluded by geometry in front of them) and scale with zoom like the tick marks themselves. An axis whose line is nearly end-on to the camera has its tick labels suppressed (its ticks would otherwise overlap near the origin); as an axis approaches that cutoff, labels (and minor ticks) thin out first — see `_axis_density` in `docs/rendering.md`. Show Edges renders the full triangulation wireframe via `GL_POLYGON_OFFSET_FILL` on the solid pass (pushes fill surfaces away from camera), then draws edges at true depth in a second pass — avoids z-fighting on coplanar faces while keeping hidden edges correctly occluded. Show Crosshairs draws four diagonal lines (the four space diagonals of a unit cube) crossing at the camera target, each extending `camera.distance * 2.5 / 12`, colored with the current theme's `axes_color` (same as the negative-axis lines/ticks) rather than a fixed color, so it reads correctly against every background theme. Perspective/orthographic toggle uses `camera.orthographic`, persisted in QSettings. Also directly clickable in the viewport itself: `Viewport._persp_btn`, a small icon button fixed in the upper-left corner (`view-perspective.svg`/`view-orthographic.svg`, `resources/icons/`, recolored white via `_recolored_icon_pixmap(..., prefix="view")` for contrast against the dark translucent button background — same treatment as the debug-pause overlay), showing the *current* mode and toggling to the other on click. `Viewport.perspective_toggled` (new/changed state) and the public `refresh_perspective_icon()` keep the button and the View menu's "Perspective" checkbox in sync in both directions — clicking the button updates the menu (`MainWindow._on_viewport_perspective_toggled`, signal-blocked to avoid re-entering `_toggle_perspective`) and toggling the menu/shortcut/restored-preference path updates the button (`_toggle_perspective` calls `vp.refresh_perspective_icon()` after flipping `camera.orthographic`).

**Restoring settings to checkable actions**: wrap `setChecked()` in `blockSignals(True/False)`, then call the handler explicitly — avoids double-invocation (signal + explicit call) and ensures the handler fires even if the stored value matches the default. Pattern used in `_restore_settings`:
```python
self._act_perspective.blockSignals(True)
self._act_perspective.setChecked(perspective)
self._act_perspective.blockSignals(False)
self._toggle_perspective(perspective)
```

**New tabs only need editor settings**: `FileTab` contains only the `CodeEditor`. There is no per-tab viewport or camera. `_new_document()` and `_create_and_add_tab()` call `_apply_preferences_to_tab(tab, font, indent, ...)` to inherit editor font/indent/guide settings and `_apply_word_wrap_to_tab(tab)` for word wrap. The `hasattr(self, '_act_word_wrap')` guard in `_new_document()` covers the case where it's called during `__init__` before `_setup_menus()` finishes.

**`FileTab` vs `DocumentTab`**: the old `DocumentTab` owned a `Viewport`, `ConsoleWidget`, `AnimatePane`, and tools strip. These are now window-level singletons on `MainWindow`. `FileTab` keeps only: `editor`, `file_path`, `is_modified`, `root_scope`, `_last_text`, `_last_cursor`, `_suppress_text_undo`.

**Cross-window sync**: `DocumentManager` (singleton in `window/document_manager.py`) tracks all open editors keyed by resolved file path. `_create_and_add_tab` calls `register(path, editor)`; `_close_tab` / `_write_file` call `unregister`. `_on_editor_changed` calls `broadcast_change(path, text, source_editor)` after each text change — other editors for the same file have `setPlainText(text)` called with signals blocked to avoid re-entrancy. `open_file_by_path` queries `get_current_text(path)` to seed a newly opened editor with in-memory (potentially unsaved) text from another window.

**Unsaved-changes prompt**: `_confirm_unsaved(tab)` is the single place that shows the "Save changes to X?" `QMessageBox` (Save/Discard/Cancel) for a modified tab, shared by `_close_tab` (closing one tab) and `closeEvent` (closing the window, looping over every tab). Returns `True` if it's fine to proceed (no unsaved changes, discarded, or saved successfully) or `False` if the caller should abort (Cancel, or a failed save) — `_close_tab` just returns early on `False`; `closeEvent` additionally calls `event.ignore()` so the window stays open. `MainWindow.skip_unsaved_prompts` (default `False`) is a testing-only escape hatch: when `True`, `_confirm_unsaved` returns `True` immediately without ever showing the modal, so a script that constructs a `MainWindow` directly can close tabs/the window without blocking on a dialog it has no way to click through programmatically.

## Data Viewers

Implemented in `src/belfryscad/window/data_viewers.py`. Seven viewer dialogs for inspecting evaluated data, opened from the debugger's variable context menu via `build_viewer_menu()`, or lexically from any numeric literal in the editor via `build_lexical_view_menu()`/`build_editor_menu()` (see "Lexical 'View as...'/'Edit as...'" above) — `PathViewer`/`GridViewer`/`MatrixViewer`/`AffineMatrixViewer`/`VNFViewer`/`RegionViewer` additionally support an `editable=True` constructor mode for the latter. An eighth, `ProfileViewer`, follows the same `QDialog`/`QTableWidget` skeleton but is launched differently — see below.

### Shared viewport: `Viewport` (`window/viewport.py`) + `SceneRenderer` (`engine/renderer.py`)

`VNFViewer`/`PathViewer`/`GridViewer` each embed the **same** `Viewport(QOpenGLWidget)` class the main window uses (as a thin per-dialog subclass — `_VNFViewport`, `_PathViewport`, `_GridViewport` — adding only picking and blinking-selection-marker overlays), rather than a separate simplified reimplementation. This means every main-window viewport feature — perspective/orthographic, stereo (cross-eye), spin (auto-rotate), show axes, show edges, show scale markers, show crosshairs, view presets, mouse-centered zoom — works identically in a data-viewer dialog, and `MainWindow`'s View-menu shortcuts apply to whichever one currently has focus (see "Keyboard shortcuts" below). There used to be a second, independent `_SimpleViewport` implementation duplicating most of this; it was deleted once all three dialogs were migrated (confirmed no remaining references before removal).

Each dialog constructs its own `Viewport(parent, selectable=False, pan_speed=2.0)` — two behavioral differences from the main window's viewport (which uses the defaults, `selectable=True`/`pan_speed=1.0`): `selectable=False` disables the Ctrl+click "select an AST/CSG original-id" gesture (meaningless for raw display data, and would otherwise stop Ctrl+drag from orbiting); `pan_speed=2.0` doubles right-drag pan's `mouseMoveEvent` scale factor (`cam.distance * 0.001 * self._pan_speed`) — data-viewer dialogs are typically zoomed in tighter on a single small shape than the main viewport is on a whole model, so the same 1x pan speed felt sluggish there. Everything else about mouse handling (plain-drag orbit, wheel zoom, Alt-drag light) works the same. Gizmo dragging and the debug-session busy-spinner overlay are main-window-only *features* of `Viewport` (opt-in via `set_active_tool`/`set_render_busy`/`set_debug_busy`), not main-window-only *code paths* — data viewers simply never call those methods, so they stay dormant.

`SceneRenderer` gained a **generic raw-buffer upload API**, additive alongside its existing `ColoredBody`/CSG-mesh path (`load_geometry`), for exactly this use case — geometry with no CSG/original-id/gizmo concept:
- `upload_mesh(positions, normals, color=..., backface_color=None, edge_positions=None, edge_colors=None, tri_ids=None) -> MeshBuffer` — `positions`/`normals` are `(3T, 3)` arrays, one row per triangle corner. Appends to the same `_buffers` list CSG bodies use (distinguished by `MeshBuffer.program is self._prog` vs. the separate `_mesh_prog`, branched in `_paint_scene`'s opaque pass), so camera-framing and `ray_cast` picking work generically across both. Passing `tri_ids` (one id per triangle) lets the mesh participate in `ray_cast`.
- `upload_lines(data) -> LineBuffer` / `upload_points(data) -> PointBuffer` — rows are `[x, y, z, r, g, b]`; lines drawn `GL_LINES`, points drawn `GL_TRIANGLES` (pre-tessellated markers, e.g. octahedra). Both reuse the existing `_gizmo_prog` (its shader is byte-identical to what a standalone line/point shader would be). `clear_lines()`/`clear_points()`/`clear_simple_buffers()` release them independent of `_clear_buffers()` (CSG-only).
- `pick_nearest_point(points, px, py, w, h) -> int` — screen-space nearest-point picking, wrapping the pure module-level `nearest_point_index()` (pytest-covered in `tests/test_renderer.py`, no GL/Qt dependency). Replaces three independently-duplicated nearest-point search loops (Path/Grid vertex picking, VNF vertex hover tooltip).
- `pick_nearest_segment(points, segments, px, py, w, h) -> tuple[int, np.ndarray] | None` — the line-segment analog, behind `PathViewer`/`RegionViewer`'s right-click-a-line "Add Vertex". `segments` is a list of `(i, j)` index pairs into `points`. Wraps two pure module-level helpers (also pytest-covered, no GL/Qt dependency): `_closest_point_on_segment_to_ray(a, b, ray_origin, ray_dir)` finds the closest point on segment `[a, b]` to the pick ray (standard closest-points-between-two-lines, segment parameter clamped to `[0, 1]`, ray parameter left unclamped), and `nearest_segment_index(...)` projects each segment's closest point to screen space and returns whichever is nearest the click, or `None` if none is within the pick threshold. Doing the closest-point search in 3D world space first (rather than, say, projecting the whole segment to screen space and doing 2D point-to-line distance there) means it works identically for 2D and 3D paths without any locked-plane branching — the resulting `world_point` already lies exactly on the segment in 3D, ready to insert as-is.
- `ray_cast()`/`camera_ray()` (pre-existing, used for the main window's Ctrl+click AST selection) are reused as-is for `VNFViewer`'s face picking, once its mesh buffer's `tri_ids` is populated with its own face-index array — this deleted ~60 lines of independently-duplicated vectorized Möller–Trumbore that `_VNFViewport._pick_face` used to carry.
- `Viewport` itself gained `schedule_load(fn)` (defers a geometry-load call until `initializeGL()` has run — needed since a dialog constructs its viewport and loads geometry before the widget is ever shown), a `_paint_extra(mvp)` no-op hook (subclasses override it for overlay geometry, called every frame right after the generic lines/points pass), `scroll_to_visible(pt)` (pans the camera minimally to keep a point on screen), and cached-bbox reframing (`frame_scene()` caches its bounds so "View All" works even when a dialog's geometry lives only in line/point buffers with no per-vertex CPU arrays to scan).

**`makeCurrent()`/`doneCurrent()` around post-initial-load GL mutation**: unlike `Viewport.load_geometry`/`closeEvent` (main window), the data-viewer subclasses' own GL-mutating methods — `_AffineViewport.load_matrix`, `_PathViewport.load_path`/`set_selected`, `_GridViewport.load_grid`/`set_selected`, `_VNFViewport.highlight_face`/`highlight_vertices` — must each bracket their buffer work in `self.makeCurrent()`/`self.doneCurrent()` when called from anywhere other than the initial `schedule_load` (which Qt guarantees runs with the right context current via `initializeGL()`). Without it, a GL call issued from an arbitrary Qt event handler (a table `itemChanged` commit, a row-selection change) can land on whatever *other* widget's context happened to be current at that moment instead of this one's — corrupting that other widget's rendering (observed: editing a `PathViewer` cell and pressing Tab blacked out the main window's viewport, because the commit fired while the main viewport's context was current).

Keyboard shortcuts (Cmd+0–9 views, Cmd+1–3 toggles, Ctrl+Cmd+1–3 toggles) are set directly on the `QAction` instances in the View menu, so they appear in the menu and work application-wide — meaning they fire regardless of which *window* currently has OS focus, main window or a data-viewer dialog. Each handler (`_toggle_perspective`, `_toggle_stereo`, `_toggle_spin`, `_toggle_axes`, `_toggle_edges`, `_toggle_scale_markers`, `_toggle_crosshairs`, `_set_view`) resolves its target via `MainWindow._target_viewport()`, which returns `_active_viewer_viewport()` (the active window's `_vp`, if it's a data-viewer dialog) or falls back to `self._viewport` (the main window's own). Since every viewport now has the identical `vp._renderer.<attr>` shape, every handler is a uniform two-liner.

**Every one of these actions explicitly sets `Qt.ShortcutContext.ApplicationShortcut`** (same as `_act_undo`/`_act_redo`/`_act_animate_tb` elsewhere) rather than relying on the default `WindowShortcut`. This isn't cosmetic: Qt's shortcut dispatch suppresses `WindowShortcut`-context actions owned by a widget outside the currently-modal-active top-level window whenever *any* modal widget is active (`QApplication.activeModalWidget()` is set) — independent of whether that dialog is `ApplicationModal` or `WindowModal`. This was discovered while the "Edit as..." editable data-viewer dialogs were still modal (before they were changed to non-modal, see above) — with a modal editable dialog focused, these shortcuts silently stopped firing even after switching modality flavors, which is what led to finding this general Qt behavior and adding the explicit context here as a defensive fix for any genuinely-modal dialog in the app (e.g. Preferences, Library Manager).

### ListViewer (QDialog)

`QTableWidget` with key/value columns. Recursive drill-down via context menu for nested lists/dicts. No 3D viewport.

### VNFViewer (QDialog)

`QSplitter`: `_VNFViewport` on left, `QTabWidget` (Vertices / Faces tabs) on right. Starts in the main window's current perspective mode. Read-only by default; pass `editable=True` for a Save/Cancel editing mode (see `MatrixViewer` for the shared editing convention) — **vertex positions only**, face topology (which vertices each face references) is never edited here (tracked as a follow-up in `TODO.md`: "VNF editor face editing").

- **Vertices tab**: `QTableWidget`, 0-indexed rows, X/Y/Z columns sized for 6 digits. Multi-select (extended selection). Selected vertices shown as blinking (red↔white, 250ms) axis-aligned cube markers in the viewport, ~6px screen size regardless of zoom or camera distance in both orthographic and perspective modes (`_marker_radius_for_point`, rebuilt on wheel). Hovering a highlighted vertex shows a tooltip with index and coordinates. When `editable=True`, cells are directly editable (`itemChanged`, same parse/validate/revert convention as `PathViewer`/`GridViewer`), updating `self._vnf[0][i][j]` in place and calling `_rebuild()`.
- **Faces tab**: `QTableWidget`, 0-indexed rows, single "Vertex Indices" column. Selecting a face highlights it green in the viewport (polygon offset overlay), deselects all vertices, and selects the face's referenced vertices. Never editable, in either mode.
- **Viewport**: `_VNFViewport(Viewport)`. Backfaces rendered magenta (`upload_mesh`'s `backface_color`). Face picking via `SceneRenderer.camera_ray()` + `ray_cast()` — the mesh is uploaded with `tri_ids` set to its own OpenSCAD face-index array (`VNFViewer._load_mesh`), so a ray hit resolves straight back to the face index. Clicking a face in viewport emits `face_clicked` signal, switches to Faces tab, and selects the row.
- **Vertex dragging and arrow-key nudging** (`editable=True` only): reuses `_PathViewport`/`_GridViewport`'s exact mechanism — `_pick_vertex` searches *all* mesh vertices (not just the currently highlighted set) so Cmd+dragging can grab any vertex regardless of table/face selection; the lock axis is always chosen via `_view_locked_axis(camera)` (VNF data has no 2D case — vertices are always 3D). Arrow keys nudge every vertex in `self._vert_indices` (the currently highlighted set, from a table/face selection) the same single-axis-snapped way `_PathViewport`/`_GridViewport.keyPressEvent` do.
- **`_rebuild()`** (editable only) re-triangulates and re-uploads the whole mesh via `_load_mesh` — face topology never changes in this dialog, so it's always the same faces against updated vertex positions. `_load_mesh` needed a `SceneRenderer._clear_buffers()` call added at its start (previously safe to omit, since it was only ever called once via the initial `schedule_load`) to avoid leaking/duplicating GPU buffers across repeated live-edit rebuilds. `_rebuild()` also needs its own `makeCurrent()`/`doneCurrent()` bracket (a live edit isn't guaranteed to run inside `initializeGL`, unlike the original single-call path) and explicitly re-invokes `_rebuild_highlight()` when a face is currently highlighted — `frame_scene` (which `_load_mesh` calls) already rebuilds vertex markers for a live edit, but not the separate highlight VAO, which would otherwise keep showing the *old* vertex positions for whatever face is highlighted.

### PathViewer (QDialog)

`QSplitter`: `_PathViewport` on left, vertex table on right. "Close Path"/"Bezier" checkboxes (20px left padding) and Dismiss button (20px right padding) below; in `editable=True` mode, Dismiss is replaced with Cancel/Save.

- **Vertex table**: `QTableWidget`, 0-indexed rows, X/Y/Z columns. Extended multi-select. Selecting rows highlights the corresponding vertices in the viewport. When `editable=True`, cells are directly editable (`itemChanged` parses/validates each edit, reverting non-numeric input, updates `self._path` in place, and calls `_rebuild()` to refresh the viewport preview) — no source writeback until Save, which emits `self._path` re-serialized via `committed`.
- **Viewport**: `_PathViewport(Viewport)`. Black lines at 2× width (`SceneRenderer.line_width`). Axis-aligned cube vertex markers (~6px screen size regardless of zoom, camera distance, or projection mode — `_marker_radius_for_point`, rebuilt on zoom): green for unselected, red↔white blink (250ms) for selected. Hovering a marker shows a tooltip with index and coordinates. Clicking a marker in the viewport emits `vertex_clicked`, deselects all, and selects the clicked vertex.
- **Close Path**: checkbox toggles whether the last vertex connects back to the first.
- **Bezier**: checkbox switches line rendering from straight segments to cubic Bezier curves. Every 4 points form one cubic segment (P0, C1, C2, P3) with shared endpoints between consecutive segments. Open paths require 3k+1 points; closed paths require 3k points (last segment wraps to first). Each curve is tessellated into 32 line segments.
- 2D paths (all Z=0) start in top-down orthographic; 3D paths start in perspective orbit. **Orbit-dragging is disabled entirely for 2D data** (`Viewport._orbit_enabled = not is_2d`, checked in the base class's `mouseMoveEvent`) — a locked top-down 2D view has no meaningful "other side" to orbit to, and dragging away from it only gimbal-locks/disorients (right-drag pan and wheel-zoom are unaffected).
- **Vertex dragging** (`editable=True` only, requires **Cmd held down**): Cmd+pressing directly on a vertex marker (`_pick_vertex`) starts a drag instead of arming orbit/pan — a plain drag (no Cmd) is always just camera orbit, in both 2D and 3D. `mouseMoveEvent` reprojects the cursor to world space via `SceneRenderer.camera_ray` intersected with an axis-locked plane (`_ray_plane_axis_locked(ray_o, ray_d, plane_point, lock_axis)` — generic plane-through-a-point with an axis-aligned normal) and emits `vertex_moved(index, x, y, z)` on every move — coordinates rounded to 3 decimal places (`round(..., 3)`) before emitting, so dragged coordinates don't carry the raw reprojection's long floating-point tail into the table/source.
  - **2D data**: the locked axis is always Z (`plane_point = [0,0,0]`) — 2D data is always embedded at Z=0 by `load_path`/`load_grid`, and the camera is locked top-down, so this is the only plane ever needed.
  - **3D data**: the locked axis is chosen at drag-start by `_view_locked_axis(camera)` — whichever world axis (X/Y/Z) is most nearly *parallel* to the camera's current view direction (`argmax(abs(target - eye_position))`), since a 2D mouse drag can't usefully control movement along an axis pointing into/out of the screen. The vertex is then only ever moved within the plane spanned by the other two (most-perpendicular-to-view) axes, through the vertex's own starting position (`plane_point = self._path_pts[vi]`/`self._all_pts[vi]` at drag-start) — the locked axis's original value is explicitly reasserted every move (`hit[lock_axis] = plane_point[lock_axis]`) to avoid float drift.
  - The dialog's `_on_viewport_vertex_moved` mirrors `_on_item_changed`'s update pattern — writes into `self._path`/`self._grid` (skipping the Z write for 2D points, which have no third coordinate), updates the matching table cell(s) (`blockSignals` around the `setText`, avoiding a recursive `itemChanged`), and calls `_rebuild(reframe=False)` for a live preview — so table and viewport stay in sync in both directions, same as before. `reframe=False` threads down through `_rebuild`/`load_path`/`load_grid`/`load_region`/VNFViewer's `_load_mesh` to `Viewport.frame_scene(bb_min, bb_max, reframe)`, skipping the actual `Camera.frame_bounds` re-fit (a full "View All" — recenter *and* rezoom) while still updating the cached bounds `frame_scene` keeps for the real "View All" action, and still rebuilding vertex markers at their new positions (each subclass's `frame_scene` override always does that part, reframe or not). Without this, every single drag/nudge frame — including a keyboard nudge, which shares the exact same `vertex_moved` signal/handler path as a drag — re-fit the camera to the whole shape's growing/shrinking bounding box, causing the view to visibly zoom/recenter on every move instead of just tracking the edited vertex. Immediately after the rebuild, `_on_viewport_vertex_moved` calls the pre-existing `Viewport.scroll_to_visible(pt)` (already used elsewhere to keep a newly table-selected vertex on-screen) with the vertex's new position — pans (never zooms) the camera the minimum amount needed if the moved point has strayed within 15% of the viewport edge (NDC `threshold = 0.85`), so a vertex being dragged/nudged toward the edge scrolls into view instead of disappearing, without the jarring reframe a full "View All" would cause on every move.
- **Arrow-key nudging** (`editable=True` only): `_PathViewport`/`_GridViewport.keyPressEvent` picks the same locked axis Cmd+drag would use for the current view (Z for 2D data; `_view_locked_axis(camera)` for 3D) and passes it to `_key_nudge_delta(camera, lock_axis, key, magnitude)`, which reads the camera's actual screen-right/up directions off `view_matrix()` rows 0/1. Unlike a drag (which can move freely to anywhere in the locked plane), each key press snaps to exactly one of the plane's two free axes rather than blending both — "Right"/"Left" moves along whichever free axis has the larger-magnitude component in the camera's screen-right direction (sign-matched to that component), "Up"/"Down" always uses the *other* free axis against screen-up, so a press always reads as a single changed coordinate in the table regardless of how the viewport has been orbited (for the 2D top-down camera this reduces to the original fixed ∓X/±Y mapping, verified unchanged). **Step size**: `_key_nudge_magnitude(event.modifiers())` — shared by every editable viewport's `keyPressEvent` (`_PathViewport`, `_GridViewport`, `_VNFViewport`, `_RegionViewport`) — returns `0.1` with Cmd held (`Qt.KeyboardModifier.ControlModifier`, Cmd on macOS — distinct from Cmd's other meaning of starting a vertex *drag* on mouse-press), `10.0` with Shift held, or the default `1.0` otherwise. For every currently-selected vertex, reads its live position out of `_path_pts`/`_all_pts`, adds the delta (rounded to 3 decimal places, same as drag), and re-emits `vertex_moved` — reusing the exact same signal/handler path as Cmd+dragging, so table sync and rebuild work identically. No-op (falls through to the base `keyPressEvent`) when not editable or when nothing is selected.
  - A press-then-release with no net movement (the same &lt;3px `_drag_started` threshold used to distinguish a click from a camera-orbit drag) still emits the ordinary `vertex_clicked` select instead of a move, so plain Cmd+clicking is unaffected.
  - Read-only viewers (`editable=False`) never enter this code path regardless of Cmd — pressing on a vertex behaves exactly as before (selects on release, no drag).
  - **Plane indicator**: while a drag is in progress, `_show_delta(f"Plane: {_unlocked_plane_name(lock_axis)}")` shows a bottom-center overlay label (`Viewport._delta_label` — the same mechanism the gizmo drag uses for its angle/scale readout, e.g. "Plane: XY"/"Plane: XZ"/"Plane: YZ") naming the two axes the drag is currently constrained to. `_unlocked_plane_name(lock_axis)` joins whichever two of "XYZ" aren't the locked one. Shown on press (once a vertex is picked) and hidden on release, regardless of whether the drag actually moved anything.
- **Adding/deleting vertices** (`editable=True` only) — no buttons; all via context menus and the keyboard:
  - **Right-click a line to Add Vertex, right-click a vertex to Delete it** (viewport, and not immediately after a right-button pan drag — `_context_menu_suppressed`, captured in `mouseReleaseEvent` before the drag-tracking state resets): `_PathViewport.contextMenuEvent` first checks `_pick_vertex` — a hit offers **"Delete Vertex"** for that specific point (`delete_vertex_requested(vertex_idx)` → `PathViewer._delete_vertex(vertex_idx)`, regardless of the table's current selection). Otherwise it builds the segment list (`self._path_pts` consecutive pairs, plus the wrap-around edge when `self._closed`) and calls `SceneRenderer.pick_nearest_segment` — for each segment, `_closest_point_on_segment_to_ray` finds the closest point on that 3D segment to the pick ray (a generic closest-point-between-two-lines computation, clamped to the segment), which is then screen-projected and compared to the click position, so this works uniformly for 2D and 3D paths without needing a locked-plane branch the way dragging does. On a hit, the context menu offers "Add Vertex", which inserts a point **exactly at the clicked location on that segment** via `add_vertex_requested(insert_after, x, y, z)` → `PathViewer._on_viewport_add_vertex_requested`. The wrap-around segment's "insert after" index is always `len(path) - 1`, so clicking the closing edge correctly appends at the end of the list rather than inserting before index 0.
    - **Right-drag panning vs. context menus**: on macOS, `contextMenuEvent` fires on the right mouse-button **press** (matching the platform convention of showing menus without waiting for release), not after `mouseReleaseEvent` the way it's easy to assume. Two consequences, both fixed here: (1) the base `Viewport`'s orbit/pan tracking (`_last_mouse`/`_mouse_button`, armed by `mousePressEvent`) must only be reset right before an actual `QMenu.exec()` call — resetting unconditionally at the top of `contextMenuEvent` (an earlier version of this fix) fired on *every* right-click, including blank-space ones that show no menu at all, which broke ordinary right-drag panning outright by clearing the drag state before any `mouseMoveEvent` could use it. (2) `_context_menu_suppressed` (set by the *previous* gesture's `mouseReleaseEvent`) is stale at the moment a *new* press's `contextMenuEvent` runs, since that press's own release hasn't happened yet — so `mousePressEvent` now also resets `_context_menu_suppressed = False` at the start of every new gesture, or a single right-drag-pan would permanently block every later right-click's context menu. Same fixes in `_GridViewport`/`_RegionViewport`'s `contextMenuEvent`/`mousePressEvent` below. `_RegionViewport` is a partial exception: it always shows a menu on any right-click (either "Add Vertex" or "Add Path" — see below), so unlike Path/Grid, right-drag panning can't coexist with it there at all when a menu fires on press.
  - **Right-click a table row for Delete Vertex**: `_vert_table.customContextMenuRequested` → `PathViewer._show_vert_table_context_menu` offers "Delete Vertex" for whichever row is under the cursor (`self._vert_table.rowAt(pos.y())`; no menu at all if the click lands on empty space below the last row) — mirrors `GridViewer`/`RegionViewer`'s table context menu convention.
  - **Delete/Backspace key deletes the current selection**: `PathViewer.keyPressEvent` catches `Qt.Key.Key_Delete`/`Qt.Key.Key_Backspace` at the dialog level (neither the table nor the viewport consumes those keys themselves, so the event otherwise just bubbles up here unhandled) and calls `self._delete_vertex()`. Works regardless of whether the selection was made by clicking table rows or viewport markers, since the two stay in sync (`_on_vert_table_selection` ↔ `_on_viewport_vertex_clicked`).
  - `_delete_vertex(vertex_idx: int | None = None)` is the single shared deletion path for all three of the above (an explicit index for a targeted right-click, or `None` to fall back to the table's current selection for the Delete key), fully repopulating the vertex table (`_populate_vert_table`, mirroring `GridViewer._populate_table`'s same row-count-change situation) and refusing (silently, no popup, matching the invalid-cell-edit convention) if that would leave fewer than the 2 points `_is_path` requires.
- **Bezier node types** (`editable=True`, `Bezier` checked only) — each on-curve vertex ("v0", `idx % 3 == 0` in the flat point list) carries a persistent, viewer-only **node type** controlling how dragging/nudging one of its two control-point handles affects the other:
  - **Disjointed**: handles move independently (a corner/cusp — this is how every vertex behaved before node types existed).
  - **Same Angle**: the opposite handle mirrors the dragged handle's *direction* through v0 but keeps its own prior distance from v0 — a continuous tangent, discontinuous curvature.
  - **Symmetric**: the opposite handle mirrors both direction *and* the dragged handle's new distance from v0 — a fully smooth node.
  
  Types are transient (`_PathViewport._node_types: dict[int v0_idx -> str]`), never serialized into the saved Path literal (OpenSCAD paths are plain point lists with no room for per-point metadata — same as `_selected_indices`). **Auto-detected** from existing handle geometry (`_classify_node_type`: opposite-direction-and-equidistant → symmetric; opposite-direction-only → same_angle; anything else, or a missing handle at an open path's start/end → disjointed) whenever the Bezier checkbox transitions off→on (`load_path`, comparing the incoming `bezier` argument against the viewport's own previous `self._bezier`) — never on an ordinary drag/nudge-triggered rebuild, so it can't clobber an explicit override or the type a user is actively dragging toward. **Set explicitly** via a "Bezier Node Type" submenu (checkmark shows the current type) added to the existing right-click-a-vertex menu when the clicked point is a v0 in Bezier mode, right above "Delete Vertex".
  - **Setting a type immediately snaps both handles into line with it**, rather than leaving them wherever they were until the next drag — `_snap_handles_to_node_type(v0, handle_a, handle_b, node_type)` (pure, alongside `_classify_node_type`): `same_angle` averages the two handles' *direction* through v0 only (each keeps its own existing distance — the shared axis is the normalized average of `handle_a`'s own direction and `handle_b`'s *opposing* direction, `da/len_a - db/len_b`, which is exactly `2*da/len_a` and thus idempotent when the pair is already perfectly opposite); `symmetric` also averages their *distance*, landing both handles equidistant from v0. Returns `None` (no snap — same for `disjointed`, a missing handle at an open-path boundary, a handle coincident with v0, or the degenerate case where the two handles already point in the exact *same* direction from v0, making "opposing average" ill-defined) or the pair. `_v0_handle_indices(v0_idx, n, closed)` (pure, shared with `classify_single_node`) resolves the same preceding/following handle indices used everywhere else node types are computed. `_set_node_type` emits the snapped positions as ordinary `vertex_moved` signals (one per handle) rather than mutating anything directly, so the write-back into `self._path`, the table, and the live rebuild all go through the exact same path a real drag would use; only when nothing needs to move does it fall back to a direct `refresh_markers()` (a `makeCurrent()`/`_build_point_markers()`/`doneCurrent()`/`update()` wrapper, needed here since — unlike `load_path`/`set_selected` — this can fire from an arbitrary context-menu action, not the initial `schedule_load`).
  - **Marker shapes** make a v0's type visible at a glance: Disjointed keeps the plain cube (`_cube_faces`, unchanged); Same Angle is a diamond (`_diamond_faces` — rotated square in 2D, octahedron in 3D); Symmetric is a triangle (`_triangle_faces` — equilateral triangle in 2D, tetrahedron in 3D). All three share `_cube_faces`'s existing `(r, is_2d)` offset-vector convention. Handle (v1/v2) markers are always the plain cube regardless of type — type belongs to the v0, not its handles. The mapping (`_NODE_TYPE_SHAPE_FN`) is a **module-level** dict, not a class attribute — referencing it from inside `_PathViewport`'s class body would have resolved the bare names to the class's own same-named instance methods (defined earlier in the same class body) rather than the free functions.
  - **Linked drag/nudge**: one pure helper, `_bezier_linked_moves(path_pts, closed, dragged_idx, new_pos, node_type)`, drives both `mouseMoveEvent`'s drag and `keyPressEvent`'s arrow-key nudge (via a shared `_resolve_bezier_moves` wrapper that looks up the dragged point's owning v0's type) — replacing each one's single `vertex_moved.emit(...)` call with a loop emitting once per point the helper says must move. Dragging/nudging a **v0** always rigid-translates both adjacent handles by the same delta (type is irrelevant) unless a handle doesn't exist (an open path's start/end has only one). Dragging/nudging a **v1/v2** consults the owning v0's type: `disjointed` (or no partner handle exists — the open-path-boundary case) moves only the dragged point; `symmetric`/`same_angle` also move the opposite-side handle per the formulas above.
    - **`_owning_v0_index(idx, n, closed)`** (pure, shared by `_bezier_linked_moves` and `_resolve_bezier_moves`) resolves which v0 a handle belongs to: v0 owns itself; **v1** (`idx % 3 == 1`, this v0's own forward handle) is owned by the v0 right before it (`idx - 1`); **v2** (`idx % 3 == 2`) is owned by the v0 right *after* it (`idx + 1`), *not* the v0 its own segment starts from — v2 is the incoming tangent handle for the *next* v0, symmetric to how v1 is the outgoing tangent handle for its own v0. Getting this backwards was a real, shipped bug: `_resolve_bezier_moves` originally computed the owning v0 via a uniform `dragged_idx - dragged_idx % 3`, correct for v0/v1 but silently wrong for v2 (it resolved to the v0 the segment *starts* from instead of the one v2 is actually attached to) — dragging the control point *before* an on-curve vertex always read that wrong v0's type (frequently `disjointed`, if the wrong v0 had no override), making it look permanently disjointed regardless of its real owning v0's type, while dragging the one *after* an on-curve vertex worked correctly. `partner_idx` (the *other* handle attached to the same owning v0 — `idx - 2` for a v1, `idx + 2` for a v2) was already correct throughout; only the type-lookup's own v0 resolution was wrong.
  - **Structural changes reindex, never reclassify**: deleting vertices (`PathViewer._delete_vertex`) and a plain (non-Bezier-mode) Add Vertex both build an `old_index -> new_index` map and call the new `_PathViewport.remap_node_types(index_map)` (wrapping the pure `_remap_node_types`) *before* mutating `self._path`, so an explicit override on a v0 elsewhere in the path survives the point list shifting under it — a deleted index's old node-type entry (if any) is simply dropped.
  - **Bezier-mode Add Vertex is a De Casteljau split**, not a plain point insert — right-clicking *on the curve* (not a vertex) picks against the **tessellated** curve (`_pick_bezier_segment_t`, reusing `_tessellate_bezier`'s own `steps=32` sampling so picking and rendering can never drift apart) instead of the straight v0-to-v0 line, recovering which cubic segment and curve parameter `t` the click landed nearest (coarse micro-segment from the pick result, refined by projecting the click onto that micro-segment). `_decasteljau_split(p0, c1, c2, p3, t)` (pure, standard cubic De Casteljau subdivision) returns 5 points `(a, d, f, e, c)` that splice in place of the segment's 2 old interior control points (`path[i0+1:i0+3] = [a,d,f,e,c]`, net **+3** points) — `f` is the new on-curve vertex, landing exactly on the original curve, and both resulting half-segments retrace the original exactly (the curve's visible shape doesn't change at all). `PathViewer._on_viewport_bezier_vertex_added` builds the reindex map (old `i0+1`/`i0+2` — the replaced control points — are dropped; everything from old `i0+3` onward shifts by `+3`) *before* splicing, then classifies just the new vertex `f` (`_PathViewport.classify_single_node`, the single-vertex counterpart of the off→on transition's full `_classify_all_node_types` — reused by both) without touching any other v0's already-recorded type.
  - **Deleting a single on-curve vertex (v0) in Bezier mode does its best to keep the curve's shape close to before**, rather than leaving a straight-line gap — `PathViewer._delete_vertex` special-cases `self._bezier_cb.isChecked() and len(rows) == 1 and rows[0] % 3 == 0` (any other selection — a handle, or multiple points at once — falls through to the ordinary plain-removal path) and delegates to `_delete_bezier_v0(i0)`:
    - **Interior v0** (both a preceding and following segment exist — for a closed path with more than one segment, always true): the two adjacent segments merge into one new segment, least-squares fit via `_fit_merged_segment(p0, c1, c2, v0, c3, c4, p3, samples=16)` (pure) to approximate the shape of both — `p0`/`p3` (the surviving neighboring v0s) are fixed, only their two facing control points (`NEW_C1`/`NEW_C2`) are solved for, via the closed-form linear least-squares solution to `minimize sum_k |Q_k - B(t_k; p0,C1,C2,p3)|^2` (`Q_k` sampled evenly across both original segments, `B` the standard cubic Bernstein blend — this reduces to an ordinary 2-unknown-per-coordinate normal-equations solve since the two unknowns' basis functions `3(1-t)^2t`/`3(1-t)t^2` don't depend on the point's coordinate). When the two original segments genuinely came from one undivided cubic (e.g. the reverse of a De Casteljau split), the fit recovers the original control points exactly; otherwise it's a good approximation (verified ~27x lower sum-squared-residual than the naive fallback of just keeping the two *outer*, unrelated handles unchanged, which `_fit_merged_segment` itself only falls back to if the least-squares system is singular, e.g. degenerate/coincident samples). Net **-3** points (the deleted v0 and its two innermost handles; the outer two handles' *values* are replaced by `NEW_C1`/`NEW_C2` rather than removed).
    - **Endpoint v0** (open path start/end, only one adjacent segment): no merge is meaningful — removing an entire terminal segment necessarily just shortens the curve — so the v0 and its own 2 handles on the one side that exists are dropped outright, leaving the neighboring v0 as the new path start/end (also net -3).
    - **Closed-path v0 alignment**: `_tessellate_bezier`'s closed-path indexing assumes every v0 sits at `idx % 3 == 0` starting from index 0 — if the deleted v0 happened to be (at or before) whatever old index would otherwise land at new index 0, every point after it would end up misaligned by 1 or 2. `_delete_bezier_v0` sidesteps this by walking survivors starting from `next_v0` (itself a v0, guaranteed to survive) rather than from old index 0, for the closed interior-deletion case specifically — verified against the case that actually exposed this (a 2-segment closed path, deleting the v0 originally *at* index 0).
    - Reindexing (`remap_node_types`) happens first, as with every other structural change, so an explicit override elsewhere in the path survives; then whichever of `p0`/`p3` (or the single surviving neighbor, for an endpoint deletion) still exist are *reclassified* (`classify_single_node`) from their new handle geometry — their own adjacent handle just changed value, so a previously-recorded type may no longer match.

### GridViewer (QDialog)

`QSplitter`: `_GridViewport` on left, Row dropdown + vertex table on right. Dismiss button (20px right padding) below; in `editable=True` mode, Dismiss is replaced with Cancel/Save. Detects lists of lists of points (`_is_grid`); rows need **not** be the same length — a ragged/non-rectangular grid (e.g. a cone's single-point apex row next to a wider base row) is supported, not just a rectangular one. When editable, cell edits (`itemChanged`) apply to whichever row the dropdown currently shows (`self._grid[row_idx][col_idx][j]`), refresh the viewport preview via `_rebuild()`, and — on Save — emit the whole grid re-serialized via `committed`; `_populate_table` blocks table signals during row-switch repopulation so switching rows doesn't misfire `itemChanged` for the newly-shown row's cells.

- **Row dropdown**: `QComboBox` selects which grid row to display in the vertex table. Changing the row highlights that row's vertices in the viewport.
- **"Row Points (N)" label**: shows the point count for the selected row (rows can have different counts).
- **Vertex table**: `QTableWidget`, 0-indexed rows, X/Y/Z columns. Extended multi-select. Selecting rows highlights individual vertices in the viewport.
- **Flat indexing for ragged rows**: since rows can differ in length, point index ↔ `(row, col)` conversion isn't simple `divmod` arithmetic. `_grid_row_offsets(grid)` computes cumulative per-row flat-index offsets once (`offsets[r]` = flat index of row `r`'s first point); `_grid_flat_to_rc(vi, offsets)` (a `bisect` lookup) converts a flat index back to `(row, col)`. Both `GridViewer` and `_GridViewport` share this scheme instead of assuming a uniform column count.
- **Orbit-dragging and vertex-dragging**: same rules as `PathViewer` above — `_orbit_enabled = not is_2d` disables camera orbit for 2D grids; Cmd+dragging a vertex marker (`editable=True`) reprojects to the Z-locked plane (2D) or the drag-start axis-locked plane from `_view_locked_axis` (3D), and emits `vertex_moved(flat_index, x, y, z)`. `_on_viewport_vertex_moved` converts the flat index via `_grid_flat_to_rc`, writes into `self._grid[row][col]` (including `z` for 3D grids), switches the row dropdown if the dragged vertex belongs to a different row than currently shown (reusing `_on_row_changed`'s existing repopulation, which already reflects the just-written coordinates) or otherwise updates the visible table cell(s) directly, then calls `_rebuild()`.
- **Add/Delete Row/Column context menu** (`editable=True` only, replacing the earlier button-based UI): right-clicking the vertex table (`setContextMenuPolicy(Qt.ContextMenuPolicy.CustomContextMenu)` + `customContextMenuRequested` → `_show_vert_table_context_menu`) opens "Add Row Before", "Add Row After", "Add Column Before", "Add Column After", "Delete this Row", "Delete this Column". "This row" is always whichever row `_row_combo` currently shows; "this column" is the specific table row (a grid column/point index) under the cursor (`self._vert_table.rowAt(pos.y())`) — the two column-specific actions are omitted entirely when the click lands on empty space below the last point, since there's no column to act on there. No menu at all in Bezier Patch mode -- see below.
- **Right-click a vertex in the viewport for Delete Row/Delete Column** (`editable=True` only, and not immediately after a right-button pan drag — same `_context_menu_suppressed` convention as `_PathViewport`): `_GridViewport.contextMenuEvent` offers "Delete Row" and "Delete Column" for whichever vertex `_pick_vertex` hits (no menu at all if the click misses every vertex — unlike Path/Region, a grid has no "click a line to add" gesture here). Both emit the flat vertex index (`delete_row_requested`/`delete_column_requested`) → `GridViewer._on_viewport_delete_row_requested`/`_on_viewport_delete_column_requested`, which convert it to `(row_idx, col_idx)` via `_grid_flat_to_rc`, switch `_row_combo` to that row (so the table reflects what's about to be deleted, matching `_on_viewport_vertex_clicked`), and delegate to the existing `_delete_row`/`_delete_column` (same methods the vertex table's own context menu uses, including their silent refuse-below-minimum behavior). `contextMenuEvent` also clears the base `Viewport`'s orbit/pan tracking right before `menu.exec()` (not unconditionally at the top), and `mousePressEvent` resets `_context_menu_suppressed` at the start of every new gesture — see the "Right-drag panning vs. context menus" note on `_PathViewport.contextMenuEvent` above for why. No menu at all in Bezier Patch mode -- see below.
- **Add/Delete Row/Column disabled in Bezier Patch mode**: a Bezier Patch's control net requires an exact 4x4 grid, so `GridViewer._is_bezier_mode()` (`self._face_mode_combo.currentText() == "Bezier Patch"`) gates every entry point into `_add_row`/`_delete_row`/`_add_column`/`_delete_column` -- guarded at the mutation methods themselves (the single shared root, refusing silently, same no-popup convention as the existing row/point-count minimums), and additionally at both context-menu builders (`_show_vert_table_context_menu` shows no menu at all; `_GridViewport.contextMenuEvent` checks a `_bezier_patch_mode` flag it caches from `load_grid`'s `bezier_patch` argument) so the menus don't even appear rather than silently no-opping when clicked.
  - `_add_row(before)`/`_add_column(before, col_idx)` share the same before/after logic: the new row/point is the midpoint with whichever neighbor sits in that direction (wrapping around if Row Wrap/Col Wrap is on), or — if there's no such neighbor (row 0's "before", the last row's "after", with no wrap; same for columns) — this row/point's own values nudged along the last coordinate so the new one isn't exactly coincident. For Add Column specifically, each row computes its own neighbor/insert position against its *own* length (not the clicked row's), so a ragged grid's shorter/longer rows each still get a sensible placement, and rows too short to reach `col_idx` at all are simply left unchanged.
  - `_delete_row()` refuses (silently) below the 2 rows `_is_grid` requires; `_delete_column(col_idx)` refuses entirely if removing it would empty any affected row (a grid row needs >= 1 point).
  - All four call `_sync_face_mode_combo()` first (adds/removes the "Bezier Patch" option to match whether the grid is still exactly 4x4 after the change, resetting the selection to "Grid Faces" if it no longer qualifies) and `_GridViewport.sync_row_bookkeeping(grid)` before `_rebuild()` — a small dedicated method (not folded into `load_grid`) so `_grid_rows`/`_row_offsets` (needed by `set_selected_row`, called from `_on_row_changed` right after) are correct even if `_rebuild()`'s GL-side work is a no-op because the widget hasn't been shown yet (no GL context = no-op `load_grid`, but the bookkeeping still needs to be right before `_on_row_changed` runs).
  - `_on_row_changed(row_idx, select_all=True)`'s `select_all` flag distinguishes ordinary row-switching (select-all, the default) from `_delete_row`/`_delete_column`, which pass `select_all=False` so the user ends up with *nothing* selected after a delete rather than the whole (possibly renumbered) row auto-selected as a side effect of switching to it — `Add Row`/`Add Column` still select-all as before, only Delete changes. `select_all=False` clears both the table selection and `_vp`'s viewport selection (`_vp.set_selected([])`) explicitly, rather than relying on `clearSelection()`'s `itemSelectionChanged` signal alone — if `_populate_table` happened to leave the freshly-repopulated table with no rows selected already, Qt won't fire that signal (no change to signal), which would otherwise leave the viewport's own still-blinking selection stale.
- **Viewport**: `_GridViewport(Viewport)`. Always draws row lines (red, within one row) and column lines (blue, between adjacent rows) via `SceneRenderer.upload_lines`. Row lines use each row's own length independently. Column lines/quad faces between a pair of rows use `min(len(row_r), len(row_{r+1}))` — the columns the two rows actually share. **Triangular grids** — any grid where two adjacent rows differ in length (`_grid_is_triangular`), e.g. a cone's apex-to-base taper or a triangular-number row progression (1, 2, 3, ...) — additionally get a third, diagonal line direction (green) and, in faces mode, matching mesh faces. Two pieces combine to cover every quad *and* every leftover point: (1) one diagonal per shared-column quad cell (`(row_r[c], row_{r+1}[c+1])` for every `c` in `[0, shared - 1)`, the same edge the mesh splits each quad on — e.g. a row growing from 3 points to 4 shares 3 columns, giving 2 quad cells, each with its own diagonal), plus (2) `_grid_fan_spec(len_a, len_b, col_wrap)`, which anchors at the shorter row's last shared-index point and fans out to any points beyond the shared range (e.g. all 7 edges of an 8-point base row fanning from a single apex point). A row pair sharing fewer than 2 columns (e.g. an apex row next to any other row) is *only* fan, with no quads or quad-diagonals. Fan triangle winding is chosen to match the adjoining quad triangles' orientation, so mesh shading doesn't flip at the quad/fan seam. A plain rectangular grid (all rows the same length) has no mismatch to trigger any of this, so it gets only row and column lines/quads. In faces mode, also draws triangulated quad mesh faces (yellow both sides) via `upload_mesh`. `show_edges = True` enables `GL_POLYGON_OFFSET_FILL` when a mesh is present, pushing face triangles slightly back so the skeleton lines always render in front. Vertex markers are depth-tested (occluded by mesh, `SceneRenderer.depth_test_points = True`), which — since the row/column skeleton lines sit exactly on top of the marker positions — meant markers and lines shared identical depth at every vertex, so the tie could resolve either way (markers flickering/z-fighting against the grid's own wireframe). Fixed the same way as the face/edge polygon-offset trick just above: `_render_simple_points` (in `renderer.py`) now applies its own small `GL_POLYGON_OFFSET_FILL` (`(-1.0, -1.0)`, pulling toward the camera) only while `depth_test_points` is on, so vertex markers win that tie and always render over grid/path lines while still being correctly occluded by mesh faces meaningfully in front of them. Green cubes for unselected, red↔white blink (250ms) for selected. The blinking-selection overlay is a *separate* draw call (`_paint_extra`, right after the main lines/points pass — see below), so it needed the identical `polygon_offset` bracket applied independently; `_GridViewport`/`_VNFViewport` (both `depth_test_points = True`) apply it there too, rather than fully disabling depth test the way `_PathViewport`'s `_paint_extra` does (Path never has occluding mesh faces, so a blanket disable is correct there — but doing that for Grid/VNF would let a selected vertex on the far side of the mesh incorrectly show through it). Hovering shows `[row,col]: (x, y, z)` tooltip. Clicking a vertex switches the dropdown to that row and selects the vertex in the table; clicking blank space (`vi < 0`) clears the table selection instead of the previous no-op, deselecting everything — matching `PathViewer`/`RegionViewer`/`VNFViewer`, all of which already had this behavior (the only one missing it).
- **Face mode**: dropdown, replacing the old "Faces" checkbox — "Grid Only" (skeleton lines only), "Grid Faces" (default; triangulated quad mesh behind the skeleton lines, as before), and "Bezier Patch" (a smooth bicubic Bezier surface instead of flat quads). "Bezier Patch" only appears in the dropdown at all for a grid that's exactly 4 rows of 4 points each (`GridViewer.__init__` checks this once at construction and simply omits the item otherwise, rather than adding it disabled — a 4x4 grid of points *is* a Bezier patch's control net, so any other shape has nothing meaningful to render). `_bezier_patch_mesh(cp, steps=16)` (module-level, `data_viewers.py`) treats the grid's row and column directions as the patch's two parametric directions `u`/`v` and evaluates `S(u,v) = sum_i sum_j B_i(u) * B_j(v) * cp[i][j]` over a `steps x steps` sample lattice, using the same cubic Bernstein basis (`(1-t)^3, 3t(1-t)^2, 3t^2(1-t), t^3`) as `_PathViewport._tessellate_bezier`'s per-segment curve — just applied in both directions at once (`np.einsum('ia,jb,abk->ijk', basis, basis, cp)`) rather than along one curve. Each sampled quad is split into 2 triangles with a cross-product face normal, exactly like the flat "Grid Faces" quad-splitting logic, and uploaded via the same `upload_mesh(..., backface_color=(0.9, 0.85, 0.1, 1.0))` call — so a Bezier patch looks and behaves identically to a quad-faced grid (same color, same polygon-offset-vs-skeleton-lines behavior) except the surface actually curves through the interior instead of being flat per quad. The patch always passes exactly through its 4 corner control points (the Bernstein basis is 1 at `t=0`/`t=1` and 0 for the other three), so corner-dragging behaves identically to quad-faces mode; only the interior curvature differs. Row/column wrap and the ragged-grid triangular-fan logic are irrelevant to Bezier mode (a rectangular non-ragged 4x4 grid can't be ragged, and wrapping a Bezier patch isn't implemented) — `bezier_patch` is a separate boolean passed alongside `draw_faces` into `_GridViewport.load_grid`, and the two are mutually exclusive (the dropdown only ever selects one).
- **Wrap mode**: dropdown, replacing the old separate "Col Wrap"/"Row Wrap" checkboxes — "No Wrap", "Wrap Columns", "Wrap Rows", "Wrap Both". `GridViewer._wrap_flags()` maps the selected text to the `(col_wrap, row_wrap)` booleans `load_grid` already took. "Wrap Columns" connects each row's last point back to its own first point, closing that row's line loop (and, in faces mode, the mesh) horizontally — applied per row, so it works even when rows have different lengths. "Wrap Rows" connects the last row back to the first, closing the grid vertically (limited to the columns shared between the last and first rows, same as any other adjacent row pair). "Wrap Both" combines both, forming a torus for a rectangular grid.
- 2D grids start in top-down orthographic; 3D grids start in perspective orbit.

### RegionViewer (QDialog)

`QSplitter`: `_RegionViewport` on left, Path dropdown + vertex table on right — modeled on `GridViewer`'s "index dropdown selects one sub-list" UI, combined with `PathViewer`'s per-path vertex editing (add/delete vertex, always-closed loop, 2D-only drag/nudge), since each path in a region *is* an independently-closed polygon, unlike a grid's rows, which connect to their neighbors. Read-only by default; pass `editable=True` for a Save/Cancel editing mode. **Regions are always 2D** — `_RegionViewport`'s camera is permanently locked top-down orthographic with orbit disabled (`_orbit_enabled = False` unconditionally), there is no 3D mode to switch into, unlike `PathViewer`/`GridViewer`.

- **`_is_region(v)`**: a list of `>= 1` closed 2D polygon paths, each with `>= 3` points, representing non-overlapping perimeters under **even-odd fill semantics** — a path nested inside another alternates solid/hole (e.g. three concentric circles = a central disc surrounded by a ring). Structurally identical to `_is_grid` whenever there are `>= 2` paths (a 2D-only grid *is* a list of point-lists too) — the one case that doesn't also read as a grid is a single-path region (no holes), since `_is_grid` requires `>= 2` rows. For the overlapping case, both "Edit as Grid..." and "Edit as Region..." are offered and the user picks the intended interpretation — the same precedent as a grid row also being a valid path, or a square matrix also being a valid affine transform.
- **Path dropdown**: `QComboBox` selects which path to display in the vertex table (labeled "Path:", analogous to `GridViewer`'s "Row:"). **"Path Points (N)" label** shows the point count for the selected path.
- **Vertex table**: `QTableWidget`, 0-indexed rows, X/Y columns only (regions are 2D-only, no Z column). Extended multi-select. When `editable=True`, cells are directly editable, same `itemChanged` parse/validate/revert convention as the other editors.
- **Delete Path / Add-Delete Vertex context menu** (`editable=True` only): right-clicking the vertex table opens "Add Vertex Before"/"Add Vertex After" (only when the click lands on a specific point — `rowAt(pos.y())`), "Delete this Path", and "Delete this Vertex" (also only on a specific point). "This path" is always whichever path `_path_combo` currently shows.
  - `_add_vertex(before, vertex_idx)` inserts the midpoint of `vertex_idx` and its neighbor in that direction. Unlike `GridViewer`'s Add Row/Column, every region path is an implicitly closed polygon (no "Close Path" checkbox), so the neighbor is always found by wrapping (`% len(path)`) — there's no "no neighbor" case to fall back on. `_delete_vertex(vertex_idx)` refuses below the 3 points `_is_region` requires (a polygon minimum, not `PathViewer`'s 2). `_delete_path()` refuses (silently) if that would leave 0 paths (`_is_region`'s minimum — unlike `GridViewer`'s 2-row minimum, a region may have just 1 path).
  - **No "Add Path" here, deliberately**: unlike Add Row/Column, a brand new path has no meaningful "before"/"after" position relative to an existing one — there's nothing sensible to place it next to. Instead, right-clicking the viewport (`_RegionViewport.contextMenuEvent`, gated on `editable`, not over an existing vertex, and not immediately after a right-button pan drag — `_context_menu_suppressed`) offers one of two actions, checked in this order:
    - **On a path's line**: "Add Vertex", inserting a point **exactly at the clicked location** on that segment (not a midpoint, unlike the table's Add Vertex Before/After). Segments span every path's own points, built fresh each time from `_path_offsets` (`(start+k, start+(k+1)%n)` per path, so each path's own wrap-around edge is always included — a region path never has an "open" case the way `PathViewer`'s does), and picked via the same `SceneRenderer.pick_nearest_segment`/`_closest_point_on_segment_to_ray` machinery as `PathViewer`. The hit segment's flat start index is converted back to `(path_idx, col_idx)` via `_grid_flat_to_rc`, so `add_vertex_requested(path_idx, col_idx, x, y)` → `RegionViewer._on_viewport_add_vertex_requested` can switch the Path dropdown first if the clicked line belongs to a different path than the one currently displayed (same as clicking a vertex does), then insert at `col_idx + 1` — which, for the wrap segment, is always exactly `len(path)`, correctly appending rather than inserting before index 0.
    - **Otherwise, blank space**: "Add Path", which appends a small rectangle centered at the clicked position (reprojected to world space via `_ray_plane_axis_locked` with `lock_axis=2`, then sized via `_marker_radius_for_point(self._vp, [x, y, 0.0]) * 8` so it looks reasonable at the current zoom) — a concrete starting shape the user then builds out with Add/Delete Vertex, rather than a duplicate of whatever path happens to be selected.
- **Left-click on blank viewport space deselects** all vertices — `_on_viewport_vertex_clicked(-1)` calls `self._vert_table.clearSelection()`, which — via the existing `itemSelectionChanged` → `_on_vert_table_selection` → `_vp.set_selected([])` chain — stops the blink timer and rebuilds markers as all-green, the same as any other selection change. This is a general convention across every viewer with vertex selection (`PathViewer`, `GridViewer`, `RegionViewer`, and `VNFViewer`'s face-click-driven equivalent) — `GridViewer` was the one that had drifted to an early-return no-op for `vi < 0` and needed the same fix applied here to match the others.
- **Vertex dragging and arrow-key nudging** (`editable=True` only): identical mechanism to `PathViewer`'s 2D case (always locked to the Z=0 plane, no `_view_locked_axis` branch needed since regions have no 3D mode at all) — Cmd+drag reprojects via `_ray_plane_axis_locked` with a fixed `lock_axis=2`, arrow keys use `_key_nudge_delta(camera, 2, key, magnitude)` with the same `_key_nudge_magnitude` step sizing (1 unit, 0.1 with Cmd, 10 with Shift) as `PathViewer`/`GridViewer`.
- **Fill mode**: dropdown — "Region Only" (outlines only) or "Region Filled" (default). Filled mode tessellates via `_region_fill_mesh(paths)`, which builds a `manifold3d.CrossSection(paths, FillRule.EvenOdd)` and extrudes a hairline (`1e-3`) sliver, reading the resulting `Manifold`'s mesh back as flat (`Z` forced to `0`) triangles — reusing `manifold3d`'s own fill-rule/triangulation rather than hand-rolling an even-odd polygon tessellator, the same trick `evaluator.to_renderable_bodies` already uses to render a lone 2D shape in the main viewport. The extruded prism's side-wall triangles collapse to degenerate zero-area slivers once flattened to `Z=0` (harmless — a few extra invisible triangles, not worth filtering out by face normal).
- **Viewport**: `_RegionViewport(Viewport)`. Each path drawn as its own closed line loop, cycling through `_REGION_PATH_COLORS` (6 colors) by path index so overlapping/nested paths stay visually distinguishable while editing. `show_edges = True` (same polygon-offset trick as `GridViewer`) pushes the fill mesh backward so the outlines reliably win the depth tie against it — both are coplanar at `Z=0`, so without this the outline could randomly z-fight against the fill depending on triangle rasterization order. Vertex markers use `depth_test_points` at its **default `False`** (unlike Grid/VNF) — since the camera is always locked top-down with no 3D orbit, there's no real occlusion scenario to get right, so the simpler "always draw markers regardless of depth" approach (same as `PathViewer`'s 2D case) suffices; `_paint_extra`'s blink overlay likewise just disables depth test outright rather than needing Grid's polygon-offset tie-break.

### MatrixViewer (QDialog)

`QTableWidget` only, no 3D viewport — displays a square 2x2 through 5x5 list of lists of numbers (`_is_matrix`, e.g. an affine/rotation matrix) as an NxN grid of cells. Row/column headers are 0-indexed (`_style_table_headers` for the header look, shared with `GridViewer`'s vertex table). Read-only by default (`NoEditTriggers`, non-editable `QTableWidgetItem`s) with a single Dismiss button; pass `editable=True` for `DoubleClicked`/`EditKeyPressed` edit triggers, Save/Cancel buttons, and an `itemChanged` handler that parses/validates each edit (reverting non-numeric input) and updates `self._value` in place — no source writeback happens until Save is clicked, which emits the whole re-serialized matrix (`data_viewers._format_value`) via the `committed` signal. Dialog size is computed from the table's content size (`resizeColumnsToContents` + summed column/row extents) rather than a fixed constant, since 2x2 and 5x5 need noticeably different window sizes. No overlap with `_is_grid`: a grid row is a list of *points* (2/3-number lists) — one nesting level deeper than a matrix row, which is a list of plain numbers. There *is* overlap with `_is_path`, though: a 2x2 or 3x3 matrix's rows are themselves valid 2D/3D points, so both "View as Path..." and "View as Matrix..." can appear for the same value; 4x4/5x5 rows are too long to match `_is_path`'s point-length check (2 or 3), so only "View as Matrix..." appears there.

### AffineMatrixViewer (QDialog)

`QSplitter`: `_AffineViewport` on left, matrix table (same look and, when `editable=True`, the same Save/Cancel editing convention as `MatrixViewer`'s) on right — an edit additionally calls `_AffineViewport.load_matrix()` to refresh the reference/transformed-shape preview live. Detects specifically a homogeneous **affine transform** matrix — `_is_affine_matrix`: a 3x3 (2D) or 4x4 (3D) `_is_matrix` whose bottom row is exactly the homogeneous identity row `[0, ..., 0, 1]` (within float tolerance) — as opposed to an arbitrary numeric matrix, which just gets `MatrixViewer`'s plain table. Every affine matrix also satisfies `_is_matrix` (3x3/4x4 are in its 2-5 range), so "View as Matrix..." and "View as Affine Transform..." both appear in the context menu for the same value.

- **Reference shape**: `_affine_reference_shape(n)` — a unit square centered at the origin for `n == 3` (2D), a unit cube for `n == 4` (3D); `_affine_shape_edges(n)` gives the wireframe edge index pairs (4 for the square, the standard 12-edge cube wireframe for the cube).
- **Transform math**: `_apply_affine(matrix, points)` — promotes each `(N-1)`-dim point to homogeneous coordinates (`[x, y, ..., 1]`), multiplies by the matrix, and drops the trailing `1` back off. Pure/`numpy`-only, pytest-covered independent of any Qt/GL setup.
- **Viewport**: `_AffineViewport(Viewport)`. Draws the reference shape as a gray wireframe and its transformed image as an orange wireframe (`SceneRenderer.upload_lines`), both via the shared `_cube_faces`/`_marker_radius_for_point` module-level helpers (extracted from what used to be near-identical duplicated methods on `_GridViewport` and `_PathViewport`, now used by every viewer with vertex markers: `_AffineViewport`, `_VNFViewport`, `_PathViewport`, `_GridViewport`, `_RegionViewport`). The transformed shape's corners get cube point markers too, with the **first corner marked red** rather than orange — since a plain square/cube has no other visual landmark, this is what makes reflections/orientation flips (e.g. a negative-determinant matrix) visible at a glance instead of looking identical to a non-mirrored rotation. Camera framing (`frame_scene`) covers the union of both shapes' bounding boxes. 2D matrices start in top-down orthographic; 3D matrices start in perspective orbit — same convention as `GridViewer`/`PathViewer`.
  - **`_marker_radius_for_point(vp, world_point)`**: computes each marker's world-space radius from the *specific point's* actual distance from the camera eye, not a single shared value for the whole batch — replacing the earlier `_viewport_marker_radius(vp)`, which used only `cam.distance` (the camera's target distance) for every marker regardless of where that marker actually was. That single-value approach happened to be exactly correct in **orthographic** mode (apparent size never depends on a point's eye-distance there at all — the defining property of orthographic projection — and `Camera.projection_matrix`'s ortho half-height is itself defined as `distance * tan(fov/2)`, deliberately matching perspective's apparent size at the target depth for a seamless mode toggle) and for any viewer permanently locked top-down (`PathViewer`/`GridViewer`'s 2D case, `RegionViewer` always — every vertex is coplanar with, and thus equidistant from, a camera looking straight down at that plane). It was **wrong in perspective mode** for any 3D scene where vertices aren't all sitting at exactly the target's depth (routine for an orbited/zoomed 3D `PathViewer`/`GridViewer`/`VNFViewer`, or `AffineMatrixViewer`'s 3D case): apparent size genuinely scales with each point's own eye-distance in perspective, so a marker nearer the eye than the target rendered larger than intended, and one farther rendered smaller — most noticeable as "zooming in close makes markers look bigger" when orbiting/zooming toward one end of a 3D shape rather than its center. Every marker-building loop (`_build_point_markers`/`_build_sel_markers`/`_build_vert_markers` across all five viewports) now builds one set of **unit**-scale offset triangles via `_cube_faces(1.0, is_2d)` (cheap, geometry only, computed once per rebuild) and multiplies by `_marker_radius_for_point(self, pt)` per vertex inside the loop, rather than computing one fixed radius for the whole batch up front.

### ProfileViewer (QDialog)

`QTableWidget` only, no 3D viewport — one row per `CallSiteProfile` from a `ProfileResult` (see `docs/evaluator.md`'s "Profiling" section). Unlike every other viewer here, it's launched directly from the Design menu ("Show Profile Report…", `MainWindow._show_profile_report`), not from `build_viewer_menu()`/a value's context menu — a profiling result isn't an OpenSCAD value with a console representation. `setSortingEnabled(True)` with a small `_NumericTableWidgetItem(QTableWidgetItem)` (`__lt__` override sorting by a stored numeric value) so numeric columns sort correctly rather than lexically — the first sortable table in this codebase; every other viewer's `QTableWidget` is unsorted. Default-sorted by "Self (ms)" descending. A search box above the table filters to rows whose `Name`/`Caller`/`Caller File` contain the typed substring (case-insensitive, `setRowHidden` per keystroke — the same pattern `DebuggerPane`'s variables-panel search uses). Row → `CallSiteProfile` lookup (for navigation and the context menu) is stored directly on each row's own item via `Qt.ItemDataRole.UserRole`, not a side list indexed by row — `setSortingEnabled` reorders the table's visual rows without reordering a separate Python list, so a row-indexed side list silently goes stale after any sort (a real, shipped bug here: double-clicking a call under the default sort could navigate to an unrelated row). `navigate_requested = Signal(str, int)` (double-click a row, or right-click for "Go to Call Site"/"Go to Declaration") is connected by `MainWindow` to `_on_debug_frame_selected` — the exact same open-tab-and-highlight-line handler the debugger's call stack table already uses, reused as-is since it has no debug-session dependency. The context menu's third action, "Show Caller Profiling", just sets the search box to that row's `caller_name`, reusing the same substring filter — a quick way to see everything a given call site invoked.

## Menu Structure

**File**: New / Open… / Open Recent ▶ / Close / Save / Save As… / — / Export… / — / Quit

**Edit**: Undo / Redo / — / Cut / Copy / Paste / Select All / — / Expand Selection / Contract Selection / — / Indent / Undent / Comment / Uncomment / — / Find… / Find & Replace… / — / Word Wrap (checkable) / — / Read Only (checkable; per-tab, defaults on for files opened from the library directory)

**Design**: Render / Render with Profiling / Show Profile Report… / — / Dump CSG Tree to Console / — / Flush Caches / — / Insert Primitive ▶ (Cube, Sphere, Cylinder, Cone, …) / Boolean Operation ▶ (Union, Difference, Intersection) *(behavior of Insert Primitive and Boolean Operation deferred)* / — / Use Library ▶ *(lists installed libraries; inserts `use`/`include` statement)* / Manage Libraries…

**Dump CSG Tree to Console** (`MainWindow._dump_csg_tree`): prints the resolved `CSGNode` tree from the last successful render (`self._last_csg_tree`, set in `_on_render_done` from `_RenderWorker.finished`'s `csg_tree` payload — `evaluator.csg_tree`, threaded through the worker's `finished` signal alongside `bodies`/`id_to_node`/`elapsed_ms`/`final_vp`) via `evaluator.format_csg_tree()` (`engine/evaluator.py`, next to `flatten_csg_tree`). "No CSG tree available — render first." if nothing has rendered yet.

**Render with Profiling** / **Show Profile Report…**: a dedicated, explicitly opt-in render pass (`MainWindow._render(profile=True)`, no keyboard shortcut — F6/Shift+F6 are already Render/Debug) that turns on per-call-site timing (`Evaluator(profile=True)`, see `docs/evaluator.md`'s "Profiling" section and `docs/rendering.md`). `_on_render_done` opens the report automatically as soon as a profiled render completes (whenever `profile_result` is non-`None`) — no need to separately click "Show Profile Report…" after every run; that action stays available to reopen the last result on demand (e.g. after closing the auto-opened window). It opens `ProfileViewer` (`window/data_viewers.py`) on the result of the last such run — a sortable table (`Name`/`Caller`/`Kind`/`Caller File`/`Line`/`Calls`/`Self (ms)`/`Self %`/`Total (ms)`/`Total %` — `Caller` is the enclosing module/function's name, or `<toplevel>` for a call made outside any module/function body; `Total` is cumulative time (self time plus children), default-sorted by Self time descending) with a summary line of total/resolve/generate/unattributed time and a search box (matches substrings against `Name`/`Caller`/`Caller File` only, case-insensitive, live-filtered via `setRowHidden` as you type — same pattern as the debugger's variables-panel search). Double-click or right-click a row ("Go to Call Site" / "Go to Declaration") to jump to source, reusing the same tab-open-and-highlight handler the debugger's call stack uses (`MainWindow._on_debug_frame_selected`). "No profile available — use Render with Profiling first." if nothing has been profiled yet. A plain F6 "Render" never profiles and pays zero overhead for it.

One line per node — `kind` and a compact params summary. Represents geometry, not the code that produced it: neither `children()` calls nor user-module calls get a node of their own — their resolved subtree is spliced directly into the enclosing node's children (see `docs/evaluator.md`), so e.g. a user module wrapping a `cube` shows up as just `cube(...)`, with no wrapper node named after the module. If that spliced subtree has more than one sibling, they're grouped under a `union()` line instead — display grouping only, no actual boolean merge (see `docs/evaluator.md`), so e.g. a module producing both a `cube` and a `sphere` with no explicit combinator shows `union() { cube(...); sphere(...); }`, not two unrelated-looking top-level siblings. Two things are deliberately never shown: a generated body count (removed — once `ManifoldCache` reuses a cache-hit ancestor's result, `generate_tree()` skips visiting that ancestor's descendants entirely, leaving their `.bodies` at the empty default; a "0" there wouldn't mean "no geometry," just "never visited this pass," which is actively misleading rather than merely uninformative) and several params keys that are pure bookkeeping or already represented elsewhere in the tree (`_DUMP_HIDDEN_PARAM_KEYS`: `op`/`name` duplicate the node's own `kind`; `group_sizes` is `_generate_csg`'s private re-chunking data; `color` is already visible as its own wrapping `color()` `CSGNode` when relevant, same as `translate`/`rotate`, so showing it again on every descendant leaf — needed at generate time, not a display concern — would just be noise, especially the common `color=None` case). `verts`/`tris`/`tri_arr` (`_DUMP_TESSELLATION_KEYS`) are hidden for every kind except `polyhedron`, since elsewhere (sphere, imported meshes, etc.) they're auto-generated tessellation data, not something a user wrote — a polyhedron's own points/faces are its actual authored content and stay visible. A transform's `args` param (`_resolve_transform`'s output, `{"name": ..., "args": {0: [2.0, 2.0, -1.0]}}` for `translate([2,2,-1])`) is special-cased via `_format_call_args`, rendering it as OpenSCAD call syntax directly in the parens — `translate([2.0, 2.0, -1.0])` — rather than as a Python dict literal behind an `args=` key, which hid the actual argument value behind `_summarize_param`'s own collapsing. `_resolve_sphere` includes its resolved `r` and `segs` in `params`, so `sphere(...)` shows its radius and resolved segment count (e.g. `sphere(r=5.0, $fn=16)`) instead of appearing empty. `_DUMP_KEY_RENAMES` renames the `segs` params key — the resolved segment count shared by `sphere`/`cylinder`/`circle`/`offset`/`text`'s resolve functions — to `$fn` for display, since that's what it actually represents; the underlying `params["segs"]` key itself is unchanged (`_generate_cylinder` etc. still read `"segs"`).

`_summarize_param` collapses a value to `<list of N>`/`<dict of N>` purely by item count (`max_items=6` — not "contains a nested container" or "is a numpy array": a small nested list/dict/array, e.g. `translate`'s `args={0: [1.0, 2.0, 3.0]}` or a small polyhedron's own points, is shown in full, each element recursively summarized) or a long string to `max_len=40` characters — needed since e.g. a large resolved array's default `repr()` can span multiple lines, which would otherwise break the one-line-per-node format. Booleans render as OpenSCAD's own lowercase `true`/`false` rather than Python's `repr()`-default `True`/`False`, matching the source syntax the dump otherwise mirrors.

Indentation is `"  " * (depth + 1)` for every depth beyond the root (root itself gets no leading padding) rather than the naively-expected `"  " * depth`: the console (`ConsoleWidget._append_foldable`) displays multi-line output with a "▼ "/"▶ " toggle prefix (2 display columns) on the first line only and no prefix on the rest, so without this compensating offset a depth-1 child's own indent would land in the same column the root's text starts at (right after the arrow), making it look like a sibling of the root instead of its child.

**Flush Caches** (`MainWindow._flush_caches`) also clears `self._csg_cache` (the `ManifoldCache` backing incremental Manifold rebuilds — see `docs/evaluator.md`), alongside the existing AST scope/node table and parser AST cache resets.
